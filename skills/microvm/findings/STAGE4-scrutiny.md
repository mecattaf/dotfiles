## Overall verdict

NOT READY for a first real-world test. The draft cannot complete a single run/keep birth: gen_id dies with exit 141 under the script's own set -o pipefail (CRITICAL, blocker). Even once that is fixed, the two core isolation/resource guarantees the tool exists to provide are silent no-ops: the krun annotation keys carry a wrong run.oci. prefix so cpus/ram_mib caps never bind, and --network loopback/--publish never sets krun.use_passt so the guest has no NIC at all — both the exact ERA metadata-only fail-open class the brief forbids (#8). doctor's boot-only smoke test cannot catch any of these because it asserts nothing actually bound in-guest. ### [critical] (bash) — gen_id() lines 246; consumed in build_birth_argv() line 882 (id="$(gen_id)")
- **verdict:** confirmed  |  **fix confirmed:** True
- **issue:** gen_id is `tr -dc 'a-z0-9' < /dev/urandom | head -c 10`. Under the script's mandated `set -o pipefail` (line 17), when `head -c 10` closes the pipe after 10 bytes, `tr` is killed by SIGPIPE and exits 141. pipefail makes the whole pipeline return 141, and `set -e` aborts. gen_id is the FIRST thing build_birth_argv does, so EVERY `run`/`keep` invocation dies with exit 141 before a container is ever launched. The birth path is non-functional under the script's own shell options.
- **evidence:** Reproduced 8/8 times: a script with `set -euo pipefail` and exactly this gen_id, called through a `case ... ) verb ;;` dispatch identical to main(), exits 141 every time (see /tmp/g2.sh run). Removing only `pipefail` makes it succeed (id printed, rc=0). Root cause: head closing the read end -> SIGPIPE to tr -> 141 -> pipefail surfaces it.
- **fix:** Make the pipeline SIGPIPE-immune. Either read a bounded chunk first so tr never writes into a closed pipe, e.g. `head -c 4096 /dev/urandom | LC_ALL=C tr -dc 'a-z0-9' | head -c 10`, or avoid the pipe entirely, e.g. `LC_ALL=C tr -dc 'a-z0-9' </dev/urandom 2>/dev/null | head -c 10 || true` is NOT enough under pipefail; prefer `read -r id < <(LC_ALL=C tr -dc 'a-z0-9' </dev/urandom | head -c10; echo)` or `id=$(head -c 32 /dev/urandom | base64 | tr -dc 'a-z0-9' | head -c 10)` with the source bounded. Audit any other `producer | head` pipelines for the same SIGPIPE-under-pipefail trap.
- **verify note:** CONFIRMED against source. Location/lines exact: gen_id at line 246 is `LC_ALL=C tr -dc 'a-z0-9' < /dev/urandom | head -c 10`; `set -euo pipefail` at line 17; consumed at line 882 `id="$(gen_id)"` inside build_birth_argv, which is reached by both verb_run (line 1057) and verb_keep (line 1105) via build_birth_argv 0/1. gen_id is the first substantive action in build_birth_argv, before create_worktree (884) and any podman launch.

Reproduction: I reproduced 20/20 (stronger than the claimed 8/8). The raw pipeline returns rc=141 ten times out of ten under pipefail. Root cause is exactly as stated: /dev/urandom is an infinite/fast source, head -c 10 closes the read end after 10 bytes, tr gets SIGPIPE -> exits 141, pipefail surfaces 141 as the command status, set -e aborts. Removing pipefail makes it succeed. So every run/keep invocation dies with exit 141 before a container is launched. Birth path non-functional under the script's own options. Critical severity justified.

Fix correctness: All three proposed fixes verified working (rc=0, 10-char id) over repeated runs. Fix A (`head -c 4096 /dev/urandom | LC_ALL=C tr -dc 'a-z0-9' | head -c 10`) gave 0/1000 short-string failures (4096 random bytes yields ~570 expected [a-z0-9] chars, far more than 10 — safe). Fix B (read from process substitution) works because the SIGPIPE happens inside the substitution subshell whose status does not propagate to outer set -e. Fix C (base64, bounded) works.

One caveat on completeness: the finding's advice to audit other `producer | head` pipelines is sound in principle, but the two other `| head` sites (line 307 podman --filter | head -n1; line 361 podman --version | head -n1) are NOT at real risk — their producers emit tiny finite output that head consumes before the pipe buffer fills, so the producer exits naturally with rc=0 (verified). Only the infinite/fast /dev/urandom source reliably triggers the trap. This does not weaken the core finding; it just means the audit yields no additional victims here.
