# Stage 2 — Verb-Surface Deliberation

Three lenses proposed independently, then critiqued each other, then a synthesis pass reconciled them.


## Phase A — Proposals (independent)


### Lens: MIRROR-PRIOR-ART: name and shape every verb to track microsandbox's `msb` CLI (confirmed from the local clone's clap enum in crates/cli/bin/main.rs), with shapes cross-checked against smolvm `machine` and krunvm so anyone fluent in those tools is instantly fluent in ours, and so the documented escape hatch (adopt/fork microsandbox later) is a near-rename rather than a re-learn. We diverge from `msb` ONLY where a LOCKED project decision forces it (ephemeral-by-default instead of msb's named-persistent default; podman-labels instead of msb's SQLite; --network none default instead of msb's permissive ingress; SSH-agent-forward instead of msb's in-guest sshd; no snapshot/volume/registry/MCP breadth). The tool keeps the noun-less flat verb form of `msb <verb>` (NOT smolvm's `machine <verb>` noun-prefix) because msb is the named escape hatch and flat verbs are what we graduate INTO. doctor (verb zero) is the one place msb has no analog, so its name comes from agent-sandbox's `ags doctor`.

Through the mirror-prior-art lens, microsandbox's `msb` is the canonical surface to track because it is BOTH the closest peer (libkrun microVMs for agent code) AND our documented graduation/escape hatch — so picking names a future `msb` adopter already knows makes the migration a near-rename. I confirmed msb's exact enum from the local clone (crates/cli/bin/main.rs): Run, Create, Start, Stop, List(ls), Status(ps), Metrics, Remove(rm), Exec, Copy(cp), Logs, Pull, Ssh, Inspect, Volume, Snapshot, Install. I adopt the seven that fit a single-user accident-threat-model sandbox verbatim — run, exec, list/ls, status/ps, logs, stop, remove/rm — preserving msb's `<image> -- <cmd...>` run shape and its visible aliases, since those aliases are also the smolvm/sbx vocabulary, maximizing cross-tool familiarity.

Three deliberate, LOCKED-decision divergences from msb, each minimized to stay close to the prior art: (1) Ephemeral-by-default. msb's `run` is ephemeral only when UNNAMED and persistent when NAMED; I keep exactly that mechanism but make it explicit via `--keep`/`--name`, so persistence is the labelled opt-in our spec mandates rather than a separate verb — closer to msb than smolvm's `-d/--detach` would be. (2) No snapshot/volume/registry/install/metrics/copy/create/start verbs. These are msb's platform-breadth and two-phase surface; they are out of scope (snapshot = our worktree/git's job; volumes/registry/install = state we refuse; create/start split is unneeded once run is the ephemeral one-shot). Dropping them keeps the script single-file-auditable and is itself the boundary our graduation signal names. (3) doctor and reap have no msb-CLI analog (msb hides its reaper and has no doctor), so I borrowed those two names from the adjacent prior art that DOES surface them — `ags doctor` and the claude-code-sandbox `purge`/`clean` severity ladder — while implementing reap on msb's own reap_stale_sandboxes pattern.

Every safe-by-default posture (--network none, read-only-except-worktree, label-only selection, ssh-agent forward, conservative krun.cpus/ram_mib, ordered path-safe teardown) is where we are STRICTER than msb's permissive engine defaults, per the accident-not-adversary threat model — but the names and shapes a user types stay msb-native, satisfying the lens. The relevant source confirming the surface is /var/home/tom/mecattaf/dotfiles/skills/microvm/references/inspiration/microsandbox/crates/cli/bin/main.rs and crates/cli/lib/commands/{run,exec,stop,remove,logs}.rs.

- `doctor` — `sandbox doctor [--json]`
    - Verb zero. Capability-probe the host before any launch: /dev/kvm exists and is accessible (KVM group membership), `crun --version` reports +LIBKRUN, the crun-krun handler/symlink is present, libkrun.so.1 and an ABI-compatible libkrunfw load, and `podman info` plus a `podman --runtime crun-krun run ... true` smoke test pass. Prints each check with PASS/FAIL and the validated podman/crun/libkrun versions. Exits 0 only if every gate passes.
    - default: Read-only: probes, launches nothing persistent (smoke test uses --rm). Hard-fails with an actionable fix message (e.g. `usermod -aG kvm`) rather than letting a later run fail cryptically. Distinct precondition exit code, never conflated with a workload exit code.
    - from: Name and role taken verbatim from agent-sandbox `ags doctor` (msb has no doctor analog); checks shaped by microsandbox/libkrun capability probes (krun_check_nested_virt / krun_has_feature / krun_get_max_vcpus) and sbx's documented `usermod -aG kvm` first-run failure.
- `run` — `sandbox run [--name N] [--keep] [--cpus N] [--memory MiB] [--network none|localhost] [--publish HOST:GUEST]... [--mount HOST:GUEST[:ro]]... [--rlimit nofile=N,nproc=N] [--ssh-agent] [--tty] [--timeout DUR] [--json] <image> -- <cmd...>`
    - The primary one-shot verb: create a fresh disposable microVM from <image>, create a per-sandbox git worktree on the host and bind it read-write at the workdir, run <cmd...> in the foreground (blocking like krun_start_enter), and propagate the workload's exit code verbatim. UNNAMED runs are ephemeral and auto-removed on exit; this is the default path. The flag set after `--` is the workload command (msb's exact `-- <cmd...>` shape).
    - default: --rm-equivalent is the default (locked ephemeral-by-default); --network none; all mounts read-only EXCEPT the tool-created worktree (the only :rw); --security-opt no-new-privileges + conservative --cap-drop; conservative krun.cpus/krun.ram_mib annotations stamped by the single centralized birth function; --publish is a silent no-op guarded against under --network none. Layer-1 (--rm clean exit) + Layer-2 (trap INT/TERM/EXIT calling `podman rm -f -v`) cleanup both attach here.
    - from: `msb run <image> -- <cmd>` (confirmed in crates/cli/lib/commands/run.rs) — ephemeral-when-unnamed is msb's own behavior; reinforced by smolvm `machine run` (ephemeral default) and krun's one-shot run = create+start. `--keep` mirrors msb's named-vs-unnamed persistence split as a flag rather than a separate verb.
- `exec` — `sandbox exec [--workdir DIR] [--env K=V]... [--user U] [--tty] [--timeout DUR] [--ssh-agent] <name> -- <cmd...>`
    - Run an additional command in an already-running named sandbox (auto-starts it if stopped, exactly as msb's resolve_and_start does). With no `-- <cmd>` in tty mode, attach to a default shell. This is the 'enter the workload' path for the opt-in persistent case.
    - default: Only operates on sandboxes carrying our managed-by label; inherits the parent sandbox's --network none and read-only posture (adds no new mounts). Secrets stay out of the guest: --ssh-agent forwards the host SSH_AUTH_SOCK (bind socket + set env var inside) so keys never enter the VM.
    - from: `msb exec <name> -- <cmd>` (crates/cli/lib/commands/exec.rs, auto-start semantics confirmed), name shared with smolvm `machine exec`, krunvm `start <name> <cmd>`, and crun-vm's SSH-based exec --as <user>.
- `list` — `sandbox list [-a/--all] [--json]`
    - List sandboxes managed by this tool by querying podman directly (`podman ps --filter label=sandbox.managed-by=<us>`), reconciling live state for free with no DB. Columns: id, status, ports/published-localhost, worktree, created, persist-flag. `--all` includes exited/persistent ones.
    - default: Selection is EXCLUSIVELY by our mandatory label set, never by name substring (the claude-code-sandbox anti-pattern). Read-only. `--json` / stable columns derived straight from `podman ps --format`.
    - from: msb `list` (visible_alias `ls`, confirmed in main.rs). We keep the long name `list` as primary like msb's enum, with `ls` alias (also smolvm `ls`, sbx `ls`).
- `status` — `sandbox status [name] [--json]`
    - Show status for one sandbox (or all if name omitted): running/exited, resource caps, published localhost ports, worktree path, labels. The introspection counterpart to `list`, pulling from `podman inspect --format`.
    - default: Read-only, label-scoped. `--json` (single object or array) is the machine-readable contract callers compose around; exit code distinguishes not-found from found.
    - from: msb `status` (command name `status`, visible_alias `ps`, confirmed in main.rs mapping ps::PsArgs). Aliased `ps` like msb; role overlaps smolvm `machine status` and arrakis `list`.
- `logs` — `sandbox logs [-f/--follow] [-n/--tail N] <name>`
    - Stream or tail captured stdout/stderr from a (running or exited) sandbox via `podman logs`, for post-mortem on a crashed workload.
    - default: Read-only; label-scoped so it refuses to dump logs of containers we don't manage. Tail defaults to a bounded line count.
    - from: msb `logs <name>` (crates/cli/lib/commands/logs.rs); identical name/shape to claude-code-sandbox `logs -f -n` and gh-runner's log model.
- `stop` — `sandbox stop [-f/--force] [-t/--timeout SECS] <name...>`
    - Gracefully stop running named (persistent) sandboxes — SIGTERM then SIGKILL after timeout (`podman stop`). State/worktree kept; does not remove. --force kills immediately.
    - default: Label-scoped; graceful by default (grace before kill), matching msb's SIGTERM-then-SIGKILL-after-timeout. Never silently discards a worktree with unpushed commits.
    - from: msb `stop <name...> -f -t` (crates/cli/lib/commands/stop.rs, force+timeout confirmed); same name in smolvm, sbx, arrakis, podman.
- `remove` — `sandbox remove [-f/--force] [--keep-worktree] <name...>`
    - Remove persistent sandboxes from inventory (`podman rm -f -v`) and tear down their tool-created worktree. --force stops-then-removes a running one. This is the explicit teardown for anything created with --keep.
    - default: Label-scoped. Worktree deletion is guarded by a strict path-safety check (inside our managed root, not a symlink, not repo root) and refuses/warns on unpushed commits unless forced; --keep-worktree preserves it. Mirrors smolvm's verification-gated, ordered teardown.
    - from: msb `remove` (visible_alias `rm`, confirmed in main.rs); aliased `rm` like msb / smolvm / sbx. Worktree-loss guard answers sbx's documented `rm` footgun.
- `reap` — `sandbox reap [--until DUR] [--dry-run] [--json]`
    - Label-driven cleanup backstop (Layer 3) for the crash case the --rm and trap layers miss: `podman ps -a -q --filter label=sandbox.managed-by=<us> --filter status=exited` then `podman rm -f -v`, with an optional age cut via the sandbox.created label / `--until`. Idempotent; safe to run at the top of every invocation, as msb fires its reaper on every command.
    - default: Touches ONLY containers carrying our managed-by label (never host-global state — the arrakis anti-pattern). Disarms the EXIT trap before sweeping to avoid double-cleanup (gh-runner lesson). --dry-run prints what it would remove. Skips sandboxes explicitly labelled persistent unless they are dead/orphaned.
    - from: Severity-ladder name from claude-code-sandbox's nuclear `purge`/`clean`, but modelled on microsandbox's reap_stale_sandboxes()/spawn_reaper() and gh-runner-krunvm's prefix-reaper — re-cast as label-driven per our locked design.

### Lens: SAFETY-MINIMALIST: the smallest verb set in which the safe path is the only expressible path. Ephemeral is not a mode you select, it is the only thing `run` does; persistence is a separate, louder verb that must be typed out. Cleanup is not trusted to one mechanism — `reap` is a first-class backstop verb, not a hidden internal. `doctor` is verb zero and gates everything. Every verb selects sandboxes exclusively by the mandatory managed-by label set (never by name substring), and every verb that lists is machine-readable by default. Fewer verbs, each with exactly one unambiguous meaning, so a clumsy agent (or a clumsy human) cannot accidentally express the dangerous thing.

Seven verbs, ordered by how a safety-first user meets them: doctor (gate) -> run (the safe default and the common case) -> persist/down/exec (the explicit, louder persistent path) -> ls/reap (observe and backstop). The minimalism is deliberate and load-bearing: ephemeral-vs-persistent is split across TWO VERBS (`run` vs `persist`) rather than a `--rm`/`--persist` flag pair, because a flag is exactly the thing a clumsy agent fat-fingers — making persistence a different word you must consciously type is the cheapest way to make the safe path the only easily-expressible one. `run` has no off switch for ephemerality at all. I rejected the common `create`/`start` split (krun, krunvm, smolvm): a two-phase lifecycle invites a half-born sandbox and a second thing to forget to clean, which is pure downside for a single-user accident model — podman's one-shot `run` collapses it. I rejected separate `stop`/`clean`/`purge` ladders (claude-code-sandbox) and `destroy`/`destroy-all` (arrakis), folding intentional teardown into one `down` and making the nuclear path the single first-class `reap`; the severity ladder collapses to exactly two rungs (intentional `down`, backstop `reap`) because more rungs is more surface for the wrong one. I deliberately KEPT `reap` as a user-facing verb even though it also auto-runs on every invocation — the safety-minimalist bet is that the backstop must be auditable and invokable, not hidden. `ls` is label-only and machine-readable by default, closing claude-code-sandbox's name-substring hole and satisfying the --json contract. Snapshot/restore/pause/resume/cp/logs/upload/download/config/policy/login are all dropped: snapshot/restore (arrakis) is the biggest complexity sink and unneeded for accident-containment (git/worktree does backtracking); network policy DSLs (microsandbox, sbx) are anti-exfiltration features outside our threat model; login (sbx) and config files (claude-code-sandbox) violate no-login/no-state-file. Every verb selects only by the mandatory managed-by label, fails closed, and touches only resources the tool stamped — so the surface itself encodes the blast-radius-containment threat model.

- `doctor` — `sandbox doctor [--json]`
    - Verb zero. Runs the precondition/capability probe and refuses to certify the host if any gate fails: /dev/kvm exists and is accessible (KVM group membership), crun was built with +LIBKRUN (parse `crun --version` feature string), the crun-krun handler/symlink is present, libkrun.so.1 and libkrunfw.so.<abi> are loadable and ABI-compatible, and a `--network none` smoke run of `true` under --runtime=krun succeeds. Emits the validated podman/crun-krun/libkrun versions so guest surprises are diagnosable. Every other verb invokes the same checks (cheaply) before doing work; `doctor` is the standalone, verbose, --json-able form. Pure read-only, mutates nothing.
    - default: Read-only; never launches a real workload, never touches host-global state. Fails CLOSED with a distinct exit code and an actionable message ('run usermod -aG kvm') rather than letting a later run fail cryptically.
    - from: name verbatim from agent-sandbox (`ags doctor`) and confirmed by textcortex `config`-as-introspection; check content from crun (feature_string LIBKRUN, /dev/kvm), libkrun (krun_has_feature/krun_get_max_vcpus capability probes), libkrunfw (ABI/soname check), sbx (`usermod -aG kvm` first-run failure).
- `run` — `sandbox run [--cpus N] [--memory MiB] [--rw PATH] [--ro PATH:ro] [--publish HOST:GUEST] [--network none|localhost] [--ssh-agent] [--timeout DUR] [--json] -- <cmd...>`
    - THE primary and only birth verb. Creates a fresh per-sandbox git worktree on the host (the tool makes it; never accepts a pre-made one), funnels everything through the ONE centralized spawn function that stamps the mandatory label set (sandbox.managed-by, .created, .id, .worktree) plus safe-by-default isolation flags, then `podman run --rm --runtime=krun` in the FOREGROUND. The container is unnamed and ALWAYS --rm: ephemeral is not a flag, it is the only behavior this verb has. On clean exit --rm removes it (layer 1); an INT/TERM/EXIT trap calls `podman rm -f -v <id>` for the interactive-crash case (layer 2), disarming itself first (`trap '' EXIT`) to avoid double-cleanup. Propagates the workload's exit code verbatim, reserving distinct codes for setup (125) vs exec (126) vs not-found (127) vs doctor/precondition failure. Resource caps go through krun.cpus/krun.ram_mib annotations (validated integers, clamped to >128 MiB / <=16 vCPU) not cgroup flags.
    - default: --network none, mounts read-only except the single tool-created worktree (bound rw with :Z), --security-opt no-new-privileges, conservative --cap-drop, small caps (1 cpu / 512 MiB) unless raised, destructive git guarded inside the guest, ssh-agent forwarded only on explicit --ssh-agent so keys never enter the guest. There is NO way to make `run` persist — that requires a different verb.
    - from: `run` = ephemeral one-shot from microsandbox (unnamed `msb run` auto-removed), smolvm `machine run`, ERA `vm temp`, gh-runner-krunvm microvm_run; one-shot run=create+start collapse from podman/crun; worktree-creation from agent-sandbox git.rs + packnplay; --network none default from sbx Locked Down / Codex `read-only`; trap-disarm-first from gh-runner-krunvm.
- `persist` — `sandbox persist [--cpus N] [--memory MiB] [--rw PATH] [--ro PATH:ro] [--publish HOST:GUEST] [--network none|localhost] [--ssh-agent] --name NAME -- <cmd...>`
    - The explicit, separate, louder opt-in to a surviving sandbox — deliberately a DIFFERENT verb from `run` so the ephemeral default can never be accidentally turned off by a stray flag. Uses the SAME centralized spawn function and SAME mandatory labels, but additionally stamps a sandbox.persist=true label and a sandbox.name label, OMITS --rm, and detaches (-d) so it survives the CLI process. Requires an explicit --name (idempotent recreate via --replace). Prints the parseable localhost endpoint(s) so ssh/Caddy/Tailscale compose around it. This is the only path that produces something `reap` must later clean.
    - default: Inherits every safe-by-default isolation flag from the spawn function (network none, ro mounts, caps). Prints a LOUD warning that this sandbox survives and is the user's responsibility to `down`/`reap`. The persist label is what scopes the reaper, so a persistent sandbox is always discoverable and never invisible. Still cannot mount anything rw except the tool-created worktree.
    - from: explicit-persistence-as-opt-in is the dominant prior-art pattern (awesome-agent-sandboxes, microsandbox NAMED vs unnamed split, crun-vm `--persistent` with loud warning); detached background birth from microsandbox `create` / smolvm `--detach` / sbx `create`; loud-warning-on-persist from crun-vm.
- `ls` — `sandbox ls [--all] [--json]`
    - List every managed sandbox by querying podman EXCLUSIVELY on the mandatory label (`podman ps --filter label=sandbox.managed-by=<us>`), never by name substring. Stable columns by default (id, status, persist, ports, created, worktree) and --json for machine consumption, derived straight from `podman ps --format`. --all includes exited/orphaned records. Read-only; this is also how a human or agent discovers what `reap` would act on.
    - default: Read-only, label-scoped: it can only ever see sandboxes the tool itself stamped, so it can never report or implicate an unrelated container. Machine-readable is the default, not an opt-in.
    - from: `ls` name from sbx and microsandbox (`msb ls`); stable-columns/--json contract from microsandbox `status --format json` and sbx `ls` column set; label-only selection is the explicit fix for claude-code-sandbox's name-substring weakness.
- `down` — `sandbox down <name> | --all`
    - Graceful, intentional teardown of a PERSISTENT sandbox the user created with `persist`. Selects by sandbox.name label (or --all over the persist label), sends SIGTERM then SIGKILL after a timeout (`podman stop` then `podman rm -f -v`). Before removing, guards the worktree: refuses to discard a worktree with unpushed/uncommitted commits unless forced, so a clumsy `down` cannot silently lose work. This is the everyday, expected cleanup; `reap` is the backstop for when this never ran.
    - default: Graceful by default (SIGTERM grace window before kill). Will NOT silently destroy a worktree with unpushed commits. Label-scoped so it can only stop things the tool manages. Distinct from the nuclear `reap`.
    - from: `down`/`stop` as the gentle, intentional teardown from arrakis `stop`, smolvm `machine stop`, microsandbox `stop`; the unpushed-commits guard from sbx (`rm` drops unfetched commits footgun) and claude-code-sandbox shadow-repo motivation; graceful-then-kill timeout from arrakis reapProcess / microsandbox.
- `reap` — `sandbox reap [--all] [--older-than DUR] [--json] [--dry-run]`
    - The first-class label-driven BACKSTOP — the whole reason it is a named verb rather than a hidden internal is that it is the safety net for the cases the other two layers miss (SIGKILLed `run`, crashed CLI, a `persist` sandbox the user forgot, a VMM segfault that bypassed --rm). Runs a cheap reconcile at the TOP of every other invocation too (so the field stays clean without a daemon), but is also invokable directly. Lists by the mandatory label, hard-kills/removes any managed sandbox whose container is dead/exited/orphaned (`podman rm -f -v`), and with --older-than uses the sandbox.created label as an age GC. Path-safety-gated: any worktree rm is refused unless the path is non-empty, strictly inside the managed root, not the root, and not a symlink. --dry-run prints what it would reap.
    - default: Operates ONLY on containers carrying our managed-by label — never host-global state (the explicit anti-pattern from arrakis wiping all tap*/br0). Idempotent. rm -rf is gated by a strict is-safe-path check (mirrors smolvm). --dry-run available so the nuclear action can be previewed. Because podman labels are the only source of truth, this needs NO state file/DB to reconcile.
    - from: `reap`/backstop concept from gh-runner-krunvm prefix reaper (relabelled to label-driven), microsandbox spawn_reaper-on-every-invocation, smolvm cleanup_orphaned_ephemeral_vms with PID+path verification; severity-ladder (down < reap) from claude-code-sandbox stop<clean<purge; age GC from podman `container prune --filter until=` and microsandbox --max-duration; path-safety guard from smolvm is_safe_cache_path.
- `exec` — `sandbox exec <name> [-it] -- <cmd...>`
    - Run an additional command inside an already-running PERSISTENT sandbox (selected by sandbox.name label), for debugging/inspection. For an ephemeral `run`, there is intentionally nothing to exec into — the whole point is one foreground workload that dies. Passes arbitrary agent-written code via base64 over `-- bash -c` to dodge quoting bugs. Does not weaken any isolation flag the sandbox was born with.
    - default: Only attaches to label-managed, already-running persistent sandboxes; cannot create or resurrect one (no auto-start that would silently extend a sandbox's life). Inherits the sandbox's existing isolation; adds no new mounts or network. Keys still never enter the guest (ssh-agent forward only).
    - from: `exec` from podman/microsandbox/smolvm/sbx; base64 command-transport trick from ERA; 'enter the workload vs the host' discipline from crun-vm; no-auto-start choice diverges deliberately from microsandbox's resolve_and_start to keep lifetimes explicit.

### Lens: PODMAN-MENTAL-MODEL: every verb is the one a podman/docker user would reach for first. The tool is a thin disciplined wrapper over `podman run --runtime=krun`, so its surface deliberately echoes podman's own verb set (run/exec/ps/rm/logs/stop/inspect), with only the additions podman lacks for this niche (doctor as verb-zero, reap as the label-driven backstop, and an explicit ephemeral/persist split expressed as flags on run rather than new nouns). Where a podman verb has a more colloquial docker alias, the docker name is offered as an alias so muscle memory from either tool lands. Anything podman already does well is passed through, not reinvented.

Through the podman-mental-model lens the design goal is zero surprise: a podman/docker user should guess our verbs correctly on the first try. So the spine is podman's own surface, used as near-passthroughs: run, exec, logs, ps, inspect, stop, start, rm, cp. We wrap `podman run --runtime=krun` directly, so `run` is the obvious primary verb and `--runtime` semantics (create+start in one shot) map cleanly onto our ephemeral --rm default. The ephemeral/persist split is expressed the podman-native way — as a flag on run (--keep, stamping a persistence label) plus the natural start/stop/rm lifecycle that flag unlocks — rather than inventing a new noun like ERA's `temp` or krunvm's split create. This keeps persistence an explicit opt-in without growing the verb count.

Only three things are NOT pure podman, because podman genuinely lacks them for this niche: (1) `doctor` as verb-zero, taken from agent-sandbox, because podman has no krun/KVM precondition gate and late cryptic failures are the cautionary tale across the prior art; (2) `reap` as the label-driven backstop, because podman is daemonless and a SIGKILLed `podman run` orphans an exited-but-registered container — the exact gap microsandbox/smolvm/gh-runner-krunvm built reapers for, which we get cheaply by keying on our mandatory labels instead of a DB; and (3) the worktree/unpushed-commit guards bolted onto rm/reap, because the tool creates the worktree and must not let clumsy agent code (or our own cleanup) destroy unpushed work.

Aliases (ls for ps, delete for rm) are offered so muscle memory from either docker or podman lands. I deliberately did NOT surface krunvm's `changevm`/`config`, arrakis's snapshot/pause/resume, microsandbox's volume/snapshot/install, or any network-allowlist DSL — those are out of our accident-not-adversary threat model and would break the single-file-auditable graduation discipline. Every safe-default is fail-closed (--network none, ro mounts, conservative caps, label-gated selection) precisely where the engine is fail-open, and selection is exclusively by our mandatory label set — never by name substring, which is the documented weakness in claude-code-sandbox and sbx.

- `doctor` — `sandbox doctor [--json]`
    - Verb-zero precondition gate. Checks /dev/kvm exists and is accessible (KVM group membership), `crun --version` reports +LIBKRUN, the crun-krun handler/symlink is present, libkrun.so.1 and libkrunfw are loadable and ABI-compatible, and runs a one-shot `podman --runtime=krun run --rm <base> true` smoke test. Prints validated podman/crun/libkrun versions. Exits non-zero with an actionable message per failed check; emits machine-readable status under --json.
    - default: Read-only: inspects host capability and never mutates anything or launches a real workload (only a `true` smoke container). Refuses nothing, breaks nothing; pure diagnosis.
    - from: agent-sandbox `doctor` (verb name lifted verbatim) + crun-vm/sbx KVM-access preflight; capability-probe pattern adapted from libkrun krun_check_nested_virt/krun_has_feature.
- `run` — `sandbox run [--name N] [--keep] [--cpus N] [--memory MiB] [--publish HOST:GUEST]... [--network none|loopback] [--mount HOST:GUEST[:ro]]... [--workdir DIR] [--env K=V]... [--ssh-agent] [-it] [--json] <image> [-- <cmd...>]`

## Phase C — Synthesis (RECOMMENDED VERB SURFACE)


Recommended surface: ELEVEN verbs -- doctor, run, keep, start, exec, logs, ls, inspect, stop, rm, reap -- a thin disciplined wrapper over `podman run --runtime=krun` where podman labels are the only source of truth (no daemon, no state file). The spine is the unanimous consensus: doctor is verb-zero (read-only, fail-closed, distinct precondition exit code, actionable `usermod -aG kvm`) and runs cheaply before every other verb; reap is a first-class label-driven Layer-3 backstop that also reconciles at the top of every invocation; every verb selects EXCLUSIVELY by the mandatory sandbox.managed-by label set (never name substring); safe-by-default isolation is uniform (--network none, read-only-except-the-one-tool-created-worktree, no-new-privileges, conservative clamped krun.cpus/ram_mib, ssh-agent forwarding so keys never enter the guest); listing is machine-readable by default; the tool CREATES the worktree and all worktree removal is path-safety-gated AND unpushed-commit-guarded. The two genuine disagreements are resolved by post-deliberation weight rather than by the original three-way split. (1) PERSISTENCE: a separate verb `keep`, not a `--keep` flag on run. SAFETY-MINIMALIST's split-verb argument is the strongest single idea for an accident-not-adversary model (a different WORD cannot be fat-fingered the way msb's stray --name flips ephemeral->persistent), PODMAN explicitly conceded to it, and the locked spec permits an opt-in verb -- so `run` is now UNCONDITIONALLY ephemeral (--rm, no off switch) and `keep --name` is the louder, labelled, warned persistence path. The name `keep` (PODMAN's proposal) is chosen over the coined `persist` to preserve the msb-graduation near-rename. (2) `start` IS included (MIRROR's late correction): without it a stopped kept sandbox can only be destroyed-and-recreated, orphaning its worktree -- the precise footgun worktree-per-sandbox exists to prevent; this is the concrete bug in both no-start designs. exec deliberately does NOT auto-start (2-of-3, diverging from msb's resolve_and_start) -- with `start` present, refusing silent resurrection costs nothing and keeps lifetimes explicit. logs and inspect both survive (rejecting SAFETY's drops): logs is the only post-mortem for an always---rm crash, and inspect is the single-object --json contract the ssh/Caddy compose story reads. REJECTED: `cp` (mount-surface-adjacent egress that the worktree/git-push channel already covers), a flag-based persistence, a second introspection verb, standalone `create`, and the whole snapshot/volume/registry/install/login/config/network-DSL family -- dropping them IS the single-file-auditability graduation discipline. Build-stage must resolve the worktree creation/branch contract, the no-lock concurrency behavior of reap-on-every-invocation, --network loopback/--publish-to-localhost semantics under krun, and the exact libkrunfw ABI acceptance matrix for doctor."


### Recommended verbs


#### `doctor`
- **signature:** `sandbox doctor [--json]`
- **behavior:** Verb zero, run before any launch and cheaply at the top of every other verb. Capability-probes the host: /dev/kvm exists and is accessible (KVM group membership), `crun --version` reports +LIBKRUN, the crun-krun handler/symlink is present, libkrun.so.1 and an ABI-compatible libkrunfw load, `podman info` succeeds, and a `podman --runtime crun-krun run --rm --network none <base> true` smoke test passes. Prints each gate PASS/FAIL with the validated podman/crun/libkrun versions.
- **safe default:** Read-only: probes only, launches nothing persistent (smoke test is --rm --network none). Fails CLOSED with an actionable fix message (e.g. `usermod -aG kvm`) and a DISTINCT precondition exit code never conflated with a workload exit code. Exits 0 only if every gate passes.
- **provenance:** Name and role verbatim from agent-sandbox `ags doctor` (crates/ags/src/cmd/doctor.rs); msb has no analog. Check content from crun +LIBKRUN feature string, libkrun capability probes (krun_check_nested_virt/krun_has_feature/krun_get_max_vcpus), libkrunfw ABI/soname check, and sbx's documented `usermod -aG kvm` first-run failure. UNANIMOUS across all three lenses, no daylight.

#### `run`
- **signature:** `sandbox run [--cpus N] [--memory MiB] [--network none|loopback] [--publish HOST:GUEST]... [--mount HOST:GUEST[:ro]]... [--env K=V]... [--workdir DIR] [--ssh-agent] [-it] [--timeout DUR] [--json] <image> [-- <cmd...>]`
- **behavior:** The primary one-shot verb and the only common path: create a fresh disposable microVM from <image> via `podman run --rm --runtime=krun`, create a per-sandbox git worktree on the host and bind it rw (:Z) at the workdir, run <cmd...> in the foreground (blocking), and propagate the workload exit code verbatim. UNCONDITIONALLY ephemeral: --rm is always on, there is no off switch on this verb. Persistence lives in the separate `keep` verb. Goes through the single centralized birth function that stamps the mandatory label set (sandbox.managed-by/.created/.id/.worktree) and the safe-by-default isolation flags.
- **safe default:** --rm always (locked ephemeral-by-default, no flag can flip it on this verb); --network none; all mounts read-only EXCEPT the single tool-created worktree (the only :rw, bound :Z); --security-opt no-new-privileges + conservative --cap-drop; conservative krun.cpus/krun.ram_mib annotations (clamped >128 MiB, <=16 vCPU); --publish is a guarded no-op under --network none. Layer-1 (--rm clean exit) + Layer-2 (self-disarming trap on INT/TERM/EXIT -> `podman rm -f -v`). ssh-agent forwarded only on explicit --ssh-agent so keys never enter the guest.
- **provenance:** `msb run <image> -- <cmd>` (crates/cli/lib/commands/run.rs; run.rs:224 removes unnamed/ephemeral sandboxes). Centralized birth + safe defaults converged unanimously. DIVERGES from msb's named-vs-unnamed-on-one-verb persistence: here run is purely ephemeral and persistence is a separate verb (see `keep`).

#### `keep`
- **signature:** `sandbox keep --name N [--cpus N] [--memory MiB] [--network none|loopback] [--publish HOST:GUEST]... [--mount HOST:GUEST[:ro]]... [--env K=V]... [--workdir DIR] [--ssh-agent] [-it] [--timeout DUR] [--json] <image> [-- <cmd...>]`
- **behavior:** The explicit, louder persistence verb: create a microVM exactly like `run` but OMIT --rm so the sandbox survives exit, stamping sandbox.persist=true + sandbox.name in addition to the mandatory label set. A different WORD from run (not a flag on run), so the unsafe non-default path cannot be reached by fat-fingering a flag or by a stray --name flipping ephemeral->persistent. Requires --name.
- **safe default:** Same centralized birth function and identical safe-by-default isolation as run (--network none, read-only-except-worktree, no-new-privileges, conservative clamped krun.cpus/ram_mib). Prints a LOUD warning that the sandbox survives and is the user's responsibility to `stop`/`rm`/`reap`. Still covered by the reap backstop if abandoned. SSH keys never enter the guest.
- **provenance:** Persistence-as-separate-verb is SAFETY-MINIMALIST's load-bearing contribution; PODMAN-MENTAL-MODEL CONCEDED to it and proposed the name `keep` over the coined `persist`. RECONCILED: 2-of-3 post-revision favor a separate verb; `keep` chosen over `persist` because it echoes msb/podman's named-persistence intent (and MIRROR's own --keep token) so the graduation rename to msb's named-run stays near, while still being a distinct word. The locked spec explicitly permits an opt-in VERB. MIRROR's --keep-flag position is the minority and is rejected (see rejected_verbs).

#### `start`
- **signature:** `sandbox start [-it] <name>`
- **behavior:** Re-start a previously kept, now-stopped sandbox via `podman start`, booting the microVM from its stored definition and reattaching. Exists ONLY to serve the persistent path created by `keep`; ephemeral `run` sandboxes never reach a startable stopped state.
- **safe default:** Operates ONLY on sandboxes carrying our managed-by label; reuses the original isolation flags/annotations stamped at birth, opening no new mounts or network; refuses anything lacking our label.
- **provenance:** podman/docker `start` verbatim; krunvm/smolvm/microsandbox (msb Start enum) all expose it. INCLUDED to close the concrete bug both SAFETY and PODMAN's no-start designs share: without start, a stopped kept sandbox can only be destroyed-and-recreated, ORPHANING its worktree -- the exact footgun worktree-per-sandbox exists to prevent. MIRROR's late inclusion is adopted; the create/start split is collapsed for the ephemeral path (run does it) but necessary for the persistent path.

#### `exec`
- **signature:** `sandbox exec [--workdir DIR] [--env K=V]... [--user U] [-it] [--ssh-agent] [--timeout DUR] <name> -- <cmd...>`
- **behavior:** Run an additional command in an already-running kept sandbox; in tty mode with no `-- <cmd>` attaches a default shell. The 'enter the workload' path for the persistent case. Does NOT auto-start a stopped sandbox -- it fails closed and directs the user to `start`.
- **safe default:** Operates ONLY on sandboxes carrying our managed-by label; inherits the parent's --network none and read-only posture, adds no new mounts; defaults to the workload user, not root. Refuses (fails closed) if the target is stopped rather than silently resurrecting it. ssh-agent forwards host SSH_AUTH_SOCK only on --ssh-agent so keys never enter the VM.
- **provenance:** `msb exec <name> -- <cmd>` (crates/cli/lib/commands/exec.rs). DIVERGES deliberately from msb's resolve_and_start auto-start: 2-of-3 lenses (SAFETY + PODMAN) reject auto-start for the accident-not-adversary model -- silently extending a lifetime the user thought was over is the surprise to avoid. With an explicit `start` verb present the user is not stranded, so refusing auto-start costs nothing and keeps lifetimes explicit.

#### `logs`
- **signature:** `sandbox logs [-f|--follow] [-n|--tail N] <name>`
- **behavior:** Stream or tail captured stdout/stderr from a running or exited sandbox via `podman logs`, for post-mortem on a crashed workload. Especially load-bearing because `run` is always --rm: a crashed ephemeral workload leaves no container to enter, so captured logs are the only post-mortem channel.
- **safe default:** Read-only; label-scoped, refuses to dump logs of containers we don't manage. --tail defaults to a bounded line count.
- **provenance:** `msb logs` (crates/cli/lib/commands/logs.rs); identical name/shape to podman, claude-code-sandbox `logs -f -n`. MIRROR and PODMAN hold it; SAFETY's drop is rejected -- PODMAN's argument that it is the ONLY post-mortem for an --rm'd crash is decisive.

#### `ls`
- **signature:** `sandbox ls [-a|--all] [--json]`
- **behavior:** List sandboxes managed by this tool by querying podman directly (`podman ps --filter label=sandbox.managed-by=<us>`), reconciling live state with no DB. Stable columns by default (id, status, persist-flag, published-localhost ports, worktree, created); --all includes exited/persistent. Also how a human or agent discovers what `reap` would act on.
- **safe default:** Selection EXCLUSIVELY by the mandatory label set, never by name substring (the claude-code-sandbox anti-pattern). Read-only. Machine-readable (stable columns / --json from `podman ps --format`) is the DEFAULT, not an opt-in.
- **provenance:** msb `List` with visible_alias `ls` (main.rs:59). PRIMARY name is `ls` (converging SAFETY's `ls` and PODMAN's `ps`-with-ls-alias instinct) -- shorter muscle-memory token shared across msb/sbx/smolvm/podman. UNANIMOUS in role.

#### `inspect`
- **signature:** `sandbox inspect [--json] <name>`
- **behavior:** Dump full configuration and runtime status of ONE sandbox (labels, annotations, OCIRuntime, mounts, published localhost ports, worktree path) via `podman inspect --format`. The single-object introspection counterpart to the many-row `ls`, and the --json metadata contract the ssh/Caddy compose story reads (worktree path, published localhost port).
- **safe default:** Read-only; label-gated to managed sandboxes; pure introspection, no side effects. --json emits the machine-readable single-object contract; exit code distinguishes not-found from found.
- **provenance:** podman/docker `inspect` verbatim; msb Inspect enum, krunvm/arrakis all surface it. Chosen over MIRROR's original `status` name (podman users reach for `inspect` first; avoids `ps`-means-process-list confusion). SAFETY CONCEDED and added it; PODMAN held it; this is the post-deliberation consensus. Exactly ONE detail verb is kept (rejecting PODMAN's having BOTH ps AND inspect).

#### `stop`
- **signature:** `sandbox stop [-f|--force] [-t|--timeout SECS] <name...>`
- **behavior:** Gracefully stop running kept sandboxes -- SIGTERM then SIGKILL after timeout (`podman stop`). State and worktree are kept; does not remove. Moot for ephemeral `run` (they die in the foreground). --force kills immediately.
- **safe default:** Label-scoped; graceful by default (grace before kill). Preserves the worktree (never deletes on stop). Warns rather than silently discarding a worktree with unpushed commits.
- **provenance:** `msb stop -f -t` (crates/cli/lib/commands/stop.rs); same name in podman/smolvm/sbx/arrakis. msb/podman-native `stop` chosen over SAFETY's coined `down` for the near-rename and because stop (gentle) vs reap (nuclear) is already a clear two-rung ladder. SAFETY's `down` collapsed stop+rm; here they are kept distinct (a stopped kept sandbox is restartable via `start`).

#### `rm`
- **signature:** `sandbox rm [-f|--force] [--keep-worktree] <name...>`
- **behavior:** Remove kept sandboxes from inventory (`podman rm -f -v`) and tear down their tool-created worktree. The explicit teardown for anything created with `keep`. --force stops-then-removes a running one.
- **safe default:** Label-scoped. Worktree deletion guarded by a strict path-safety check (inside our managed root, not a symlink, not repo root, non-empty) and refuses/warns on unpushed commits unless forced; --keep-worktree preserves it.
- **provenance:** msb `Remove` with visible_alias `rm` (main.rs:70-71); podman-native. Worktree-loss guard answers sbx's documented `rm` footgun and smolvm's is_safe_cache_path. UNANIMOUS in role across all three lenses.

#### `reap`
- **signature:** `sandbox reap [--until DUR] [--dry-run] [--json]`
- **behavior:** Label-driven cleanup BACKSTOP (Layer 3) for the crash cases --rm and traps miss (SIGKILLed run, VMM segfault, an abandoned kept sandbox): `podman ps -a -q --filter label=sandbox.managed-by=<us> --filter status=exited` then `podman rm -f -v`, with an optional age cut via the sandbox.created label / --until. Runs as a cheap reconcile at the top of every invocation (so the field stays clean without a daemon) AND is directly invokable so the backstop is auditable, not hidden.
- **safe default:** Touches ONLY containers carrying our managed-by label -- never host-global state (the arrakis destroyAllVMs/tap*/br0 anti-pattern). Idempotent. Disarms the EXIT trap before sweeping (gh-runner lesson). --dry-run previews. Skips healthy running sandboxes and kept ones unless dead/orphaned. Same path-guarded + unpushed-commit-guarded worktree removal as `rm`.
- **provenance:** Severity-ladder name from claude-code-sandbox `purge`/`clean` (collapsed to ONE reap), modelled on microsandbox reap_stale_sandboxes()/spawn_reaper() (fires every command, main.rs:307) and gh-runner-krunvm prefix-reaper, re-keyed to LABELS. THE SINGLE STRONGEST CONSENSUS -- first-class verb, unanimous across all three lenses.

### Rejected verbs
- `cp (copy)` — PODMAN-MENTAL-MODEL's weakest verb; SAFETY and MIRROR reject it and PODMAN conceded it would be the first dropped at the graduation line. `podman cp` into a running krun microVM is a mount-surface-adjacent ingress/egress channel that fights 'read-only mounts where possible'. The tool-created worktree (git push out) is the sanctioned artifact channel for an accident-containment model; cp is pure extra mutation surface and verb count for no threat-model gain.
- `persist` — The persistence verb IS adopted, but under the name `keep`, not the coined `persist`. `persist` exists in no prior art (msb/podman/smolvm/arrakis), maximizing safety but minimizing the zero-surprise familiarity the graduation rename depends on. `keep` is still a distinct word (satisfying SAFETY's load-bearing split-verb requirement) while echoing msb/podman named-persistence intent. PODMAN proposed this and SAFETY accepts any distinct word; the property that matters is a DIFFERENT WORD, not a flag.
- `--keep / --name as flags on run` — MIRROR-PRIOR-ART's flag-based persistence is rejected (minority post-revision: PODMAN conceded away from it). msb's own mechanism is the hazard -- a stray --name flips a run from ephemeral to persistent. For accident-not-adversary, persistence must be an unmistakable different word, not a flag a clumsy agent can fat-finger or a default that flips on stray args. The locked spec permits an opt-in verb; the separate verb makes the unsafe path unexpressible-by-accident.
- `status (as a second introspection verb) / ps duplication` — Introspection is collapsed to exactly ONE detail verb (`inspect`) plus the many-row `ls`. MIRROR's original `status` is renamed to the podman-native `inspect`; PODMAN's instinct to surface BOTH ps AND inspect is rejected -- two introspection verbs is more surface for no threat-model gain.
- `create (two-phase create/start lifecycle)` — msb ships Create AND Run AND Start. The ephemeral path collapses create+start into `run`, and the persistent path uses `keep`(create+run) + `start`. A standalone `create` (build-but-don't-run) adds a stopped-never-started state that only multiplies reap's orphan-tracking burden with no accident-model benefit.
- `snapshot / restore / pause / resume / volume / registry / pull / install / login / config / ssh / metrics / network-policy-DSL` — All out of scope, unanimous across all three lenses. Out of the accident-not-adversary threat model (snapshot/pause/metrics), violate no-state-file / no-login / no-daemon (registry/install/login/config), or are anti-exfiltration surface (network-allowlist DSL) the project explicitly does not pursue. ssh-into-guest is unneeded since ssh-agent forwarding keeps keys out and exec/run are the entry paths. Dropping these IS the single-file-auditability graduation discipline. Tailscale/remote/host concerns stay orthogonal -- NO --host flag.

### Global flags
- --json: machine-readable output on every read verb (doctor, ls, inspect, reap --dry-run); a stable single-object/array contract derived from `podman ... --format`. Locked machine-readable-output decision; on by spirit for listing verbs (stable columns are the default even without --json).
- --help/-h: per-verb usage; flag-preceded, scriptable arg parsing.
- -it: interactive TTY allocation, shared by run/keep/start/exec.
- --ssh-agent: opt-in host SSH_AUTH_SOCK forwarding (run/keep/exec) so keys never enter the guest; never on by default.
- --network none|loopback: safe default none on every birth verb; loopback only when explicitly requested; --publish is a guarded no-op under none.
- --cpus N / --memory MiB: map to conservative, clamped krun.cpus/krun.ram_mib annotations on birth verbs.
- --timeout DUR: bound foreground workload runtime (run/keep/exec).
- --force/-f: explicit affirmative override for guarded teardown (stop/rm), required to discard a worktree with unpushed commits.
- --dry-run: preview without mutation (reap; extendable to rm).
- Distinct, stable exit codes: a DISTINCT precondition-failure code (doctor and the per-verb precheck) that is NEVER conflated with a propagated workload exit code; not-found vs found distinguished on inspect; stable across releases.

### Open questions for the build stage
- Worktree creation contract: exact managed root path (e.g. ~/.local/share/sandbox/worktrees/<id>), naming scheme keyed to sandbox.id vs sandbox.name, and the branch/checkout policy when `keep`/`run` create the worktree (new branch per sandbox? detached? from which ref?). The destructive-git guard's precise refusal rules (repo root, symlink, non-empty, unpushed/uncommitted) need concrete predicates.
- reap-on-every-invocation cost and concurrency: running the reconcile sweep at the top of every verb must be cheap and must not race two concurrent invocations into double-rm or trap/reap double-cleanup. Confirm the EXIT-trap-disarm-before-sweep ordering holds under concurrent runs, since there is no state file/lock by design. This is also a named graduation signal (concurrency-safe allocation) -- decide the threshold at which bash stops sufficing.
- --network loopback semantics under krun: confirm what 'loopback' maps to (pasta/slirp4netns? host-loopback-only?) and how --publish HOST:GUEST binds predictably to localhost for the ssh/Caddy compose story without a --host flag, including behavior when none is selected.
- libkrunfw ABI/soname compatibility matrix: which exact libkrun.so.1 / libkrunfw soname versions doctor should accept vs reject on Fedora, and whether the krun_* capability probes are callable from bash (likely need a tiny helper) or must be inferred from `crun features` + version strings.

- krun.cpus/krun.ram_mib clamp bounds: confirm the concrete floor/ceiling (>128 MiB, <=16 vCPU were proposed) against libkrun's krun_get_max_vcpus and Fedora host realities.
- Base image policy: doctor's smoke test and run's default <image> -- is there a canonical Fedora base the tool assumes/pulls, and does pulling violate the no-registry-verb stance (pull-on-run is implicit, not a verb)?
