## crun (krun handler) — github.com/containers/crun

**Tier:** documentation  |  **Language:** C (the krun handler is src/libcrun/handlers/krun.c; docs in krun.1.md / crun.1.md)

**Purpose:** krun is crun running in "krun mode" (a symlink to the crun binary, also usable as podman --runtime=krun). It is a custom OCI runtime handler that, instead of running the container in host namespaces, dlopens libkrun and boots the container's rootfs as a libkrun/KVM microVM, exec-ing the workload inside the guest. This is the exact engine our project sits on.


### Verb surface

- `(invocation) krun OR podman --runtime=krun` `krun [global options] command [...]  /  podman run --runtime=krun ...` — krun is a symlink to the crun executable that flips it into krun mode. There is NO krun-specific subcommand: krun inherits crun's entire verb set verbatim (krun.1.md COMMANDS section literally says 'See crun.1 man page'). Handler is selected via the OCI annotation run.oci.handler=krun, the runtime global handler, or podman --runtime=krun.
- `create` `create [options] CONTAINER` — Create a container (runtime detaches once env is created); must be followed by start. The create half of the create/start split.
- `start` `start CONTAINER` — Start a previously-created container. Cannot be started more than once.
- `run` `run [options] CONTAINER` — Create AND immediately start a container in one shot. The verb podman uses under the hood; the one-shot ephemeral path.
- `exec` `exec CONTAINER CMD` — Exec a command in a running container. For krun guests, exec into a microVM is more constrained than a normal namespaced container.
- `kill` `kill CONTAINER [SIGNAL]` — Send signal to the container init process; defaults to SIGTERM if none given.
- `delete` `delete CONTAINER` — Remove the definition/state for a container (teardown of runtime state).
- `state` `state CONTAINER` — Output the OCI state of a container (JSON).
- `list` `list` — List known containers.
- `ps` `ps CONTAINER` — Show processes running in a container.
- `pause` `pause CONTAINER` — Pause all processes in the container (freezer).
- `resume` `resume CONTAINER` — Resume paused processes.
- `update` `update CONTAINER` — Update container resource constraints.
- `spec` `spec` — Generate a default OCI config.json.
- `checkpoint` `checkpoint CONTAINER` — Checkpoint a running container via CRIU (almost certainly N/A for microVMs).
- `restore` `restore CONTAINER` — Restore a container from a CRIU checkpoint.
- `mounts add` `mounts add CONTAINER mounts.json` — EXPERIMENTAL. Add mounts to a running container from the mounts section of an OCI config; can change without notice.
- `mounts remove` `mounts remove CONTAINER mounts.json` — EXPERIMENTAL. Remove mounts from a running container (only destination is used); can change without notice.

### Lifecycle model
krun itself carries NO lifecycle/cleanup/reaper logic of its own — it is a stateless OCI runtime. Lifecycle is owned entirely by the OCI two-phase model (create -> start, or run = both) and by whatever manager invokes it (podman --rm, podman stop). State lives under /run/crun (root) or $XDG_RUNTIME_DIR/crun (rootless); --root overrides. The microVM is bound to the krun_start_enter() call: libkrun_exec configures the VM, then krun_start_enter(ctx_id) blocks running the guest; when the guest workload exits, the call returns and the process tree unwinds. So the VM is exactly as ephemeral/persistent as the container record — there is no daemon, no background VM survives the crun process. Confirms our 'podman is the only source of truth, no state file, no daemon' design: krun gives us nothing to reap on its own; reaping is a podman-label concern.

### Isolation & flags
- Handler selection: podman run --runtime=krun, OR OCI annotation run.oci.handler=krun, OR a .krun_vm.json at image root. Annotations take precedence over .krun_vm.json. Attempting to override an already-configured global handler via the spec fails with EACCES.
- krun.cpus=NUM: vCPUs, hard max 16 (LIBKRUN_MAX_VCPUS). Default = process CPU affinity count (sched_getaffinity), capped at 16, falling back to 1.
- krun.ram_mib=NUM: guest RAM in MiB. Values <=128 are ignored (LIBKRUN_MINIMUM_RAM_MIB=128). If unset/too small, falls back to the OCI memory limit (def->linux->resources->memory->limit) if present, else 1024 MiB (LIBKRUN_DEFAULT_RAM_MIB). So podman --memory feeds RAM when krun.ram_mib is absent.
- Network DEFAULT IS NO NETWORK: networking only exists if krun.use_passt>0, which forks a passt process (passt -t all -u all --no-dhcp-dns --fd <n>) wired over a unix socketpair via krun_add_net_unixstream with DHCP client flag. Without it the guest has no NIC. Aligns with our --network none default — krun is already network-off by default.
- krun.nested_virt=NUM>0: exposes VMX/SVM to guest; warns if host /sys/module/kvm_*/parameters/nested is not Y/1.
- krun.gpu_flags=FLAGS>0: virtio-gpu/virgl; requires /dev/dri and /usr/libexec/virgl_render_server or it errors.
- krun.variant / .krun_vm.json flavor: 'sev' (AMD SEV TEE, needs libkrun-sev.so.1) or 'aws-nitro' (needs libkrun-awsnitro.so.1). Default plain libkrun.so.1.
- Root filesystem is shared into the guest via virtiofs (krun_add_virtiofs2 mounting host rootfs at guest tag, default tag /dev/root, DAX shm default 512 MiB). .krun_vm.json can set virtiofs_tag and virtiofs_shm_size. This is how host files reach the guest — mounts are virtiofs-backed.
- /dev/kvm is mandatory for all non-nitro workloads; krun errors with '`/dev/kvm` unavailable' if absent. The handler also auto-injects /dev/kvm (and /dev/sev, /dev/nitro_enclaves when those libs are present) into the OCI device cgroup allowlist and creates the device node, so you do NOT need to manually pass --device /dev/kvm.
- .krun_vm.json (at image root, path /.krun_vm.json) fields beyond cpus/ram_mib/gpu_flags/use_passt/nested_virt/flavor: kernel_path, kernel_format, initrd_path, kernel_cmdline (external kernel; else libkrunfw kernel is used), virtiofs_tag, virtiofs_shm_size.
- Annotation values are parsed as strict positive integers; a non-integer or negative value is a hard EXIT_FAILURE error.

### Standout features
- Zero new verbs: krun is literally crun with a flag, so the entire scriptable surface is podman's + crun's existing OCI verbs. Nothing bespoke to learn for the caller.
- Config is purely declarative via OCI annotations and an in-image .krun_vm.json — no krun-specific CLI flags exist. Resource sizing is annotation-driven (krun.cpus / krun.ram_mib), exactly the lever our centralized spawn function will pull.
- Network-off by default (passt is strictly opt-in) — the runtime's own default matches a safe-by-default isolation posture.
- Graceful capability fallback: dlopens libkrun/libkrun-sev/libkrun-awsnitro independently and degrades if a variant lib is missing; unknown libkrun symbols (gpu, nested_virt) are tolerated for older libkrun.
- Confidential-computing variants (SEV, AWS Nitro) selectable purely by annotation/flavor.
- Hardened against CVE-2025-24965: writes/reads config files under rootfs using safe_openat + O_NOFOLLOW because rootfs content is attacker-controlled.

### Gotchas
- krun.ram_mib values of 128 or less are silently ignored and fall through to the OCI memory limit or 1024 MiB — don't expect a tiny RAM cap to stick. The test is '<=128', so exactly 128 is ignored.
- krun.cpus is hard-capped at 16; default derives from CPU affinity, so on a big host an unset value yields up to 16 vCPUs — set krun.cpus explicitly to stay conservative.
- podman --memory feeds VM RAM ONLY when krun.ram_mib is unset; if you set both, krun.ram_mib (annotation) wins because annotations take precedence. Be deliberate so you don't fight yourself.
- No network at all unless krun.use_passt>0 — fine for our default, but 'publish to localhost' requires explicitly enabling passt and then podman port-publishing; verify the passt path works rootless before relying on it.
- exec/ps/pause into a microVM guest behave differently than a normal namespaced container; don't assume full parity with crun-on-namespaces for those verbs.
- Annotation parse errors are fatal (EXIT_FAILURE), so a malformed krun.cpus kills the run rather than warning — validate integers in our script before passing them.
- /dev/kvm is required; if KVM is missing the run dies at exec time with '`/dev/kvm` unavailable' — our doctor verb should check this up front.
- Rootfs is exposed to the guest via virtiofs of the container rootfs; 'read-only mount' semantics depend on the OCI mount/rootfs readonly flags, not on krun — krun does not add its own read-only guarantee.

### Fedora fit
Very high / Fedora-canon. crun is the default OCI runtime on Fedora and is the runtime podman uses by default; krun ships as the crun-krun subpackage and is the officially packaged path to libkrun/libkrunfw microVMs. The README explicitly demonstrates podman --runtime=/usr/bin/crun and has a Fedora section. This repo IS the upstream of the exact crun-krun binary our locked engine depends on (podman run --runtime=krun over official Fedora crun-krun + libkrun + libkrunfw). No third-party runtime involved.

### Lessons for our tool
- Our tool exposes its OWN verbs (doctor/run/persist/list/reap etc.); krun contributes NO verb names to mimic — the prior art is the OCI verb set we already get via podman. The naming inspiration to take is the clean create/start split and one-shot 'run' = create+start, which maps to ephemeral --rm.
- Drive sizing through krun.cpus and krun.ram_mib annotations (NOT bespoke flags), set inside our single centralized spawn function. Pass conservative explicit integers because defaults are generous (up to 16 vCPUs from affinity, 1024 MiB RAM).
- Respect the >128 MiB and <=16 vCPU rules: clamp/validate our --memory and --cpus inputs before stamping annotations, and prefer setting krun.ram_mib explicitly rather than relying on --memory, since annotation precedence can surprise.
- Network-off is krun's own default (passt opt-in) — so --network none is reinforced by the runtime; only wire passt + podman port-publish when a sandbox actually needs to serve to localhost, and keep that an explicit opt-in flag, not the default.
- Doctor verb (verb zero) must check exactly: /dev/kvm exists and is accessible, crun was built with +LIBKRUN (feature_string 'LIBKRUN' in the handler), crun-krun handler/symlink present, and libkrun.so.1 loadable. These are the precise failure points in the C handler (dlopen of libkrun.so.1, '`/dev/kvm` unavailable').
- Do NOT manually pass --device /dev/kvm; the krun handler auto-injects the kvm device and cgroup allow rule. Keep our isolation flags minimal and let the handler manage device exposure.
- Because krun has no daemon/state/reaper of its own, our three-layer cleanup (--rm + trap + label-driven reap) is the right and only place for lifecycle: podman labels remain the single source of truth, validated by krun adding nothing to track.
- Confidential/variant features (SEV, nitro, gpu, nested_virt) are out of scope for a single-user accident-threat-model code sandbox — explicitly ignore krun.variant/gpu/nested_virt to keep the script auditable; mention they exist but don't surface them.
- If we ever want baked-in VM defaults independent of CLI, .krun_vm.json in our base image is an option — but annotations are simpler and override it, so prefer annotations from the script and skip the JSON file to avoid two sources of truth.
- Validate integer annotation values in bash before passing: krun treats malformed krun.cpus/krun.ram_mib as a fatal error, so bad input crashes the run instead of degrading.

## libkrun (containers/libkrun)

**Tier:** documentation  |  **Language:** Rust (library), C public API header, C examples

**Purpose:** A dynamic library that embeds a minimal VMM (KVM on Linux, HVF on macOS/ARM) so a host program can run a process inside a lightweight microVM. It is the engine UNDER our stack: crun's krun handler calls into libkrun, which is what `podman run --runtime=krun` ultimately drives. It is not a tool with verbs but a config-then-launch C API.


### Verb surface

- `krun_create_ctx / krun_free_ctx` `() -> ctx_id; (ctx_id)` — Allocate/free an opaque configuration context. Everything else mutates this context before launch. Analogous to 'open a config builder'.
- `krun_set_vm_config` `(ctx_id, num_vcpus:uint8, ram_mib:uint32)` — THE resource-cap verb. Sets vCPU count and RAM in MiB. This is exactly what podman's krun.cpus / krun.ram_mib annotations feed into. Example uses (4 vcpus, 4096 MiB).
- `krun_set_root` `(ctx_id, root_path)` — Set a host directory as guest / via virtio-fs (the FS-passthrough soft spot). Convenience wrapper over krun_add_virtiofs3 with tag /dev/root.
- `krun_add_virtiofs / krun_add_virtiofs2 / krun_add_virtiofs3` `(ctx_id, tag, host_path[, shm_size][, read_only:bool])` — Add a virtio-fs share of a host dir under a guest tag. v3 adds the read_only flag — the ONLY in-library knob for read-only passthrough. No path-jail; see gotchas.
- `krun_add_disk / krun_add_disk2 / krun_add_disk3` `(ctx_id, block_id, disk_path, [format], read_only:bool, [direct_io, sync_mode])` — Attach a raw/qcow2/vmdk disk image as a block device, with read_only and direct_io flags. add_disk2/3 carry a long SECURITY NOTE about format probing.
- `krun_set_root_disk_remount` `(ctx_id, device, fstype, options)` — Boot a block-device rootfs (e.g. /dev/vda1, ext4) instead of virtiofs, via a dummy virtiofs that runs init then switch_root. Mount options are a plain string.
- `krun_add_net_unixstream / krun_add_net_unixgram / krun_add_net_tap` `(ctx_id, path|fd, mac, features, flags)` — Add a virtio-net interface backed by passt/gvproxy (unix socket/fd) or a host tap. ANY of these DISABLES the default TSI networking. Interfaces become eth0, eth1... in add order.
- `krun_set_port_map` `(ctx_id, ["host_port:guest_port", ...])` — TSI-only port forwarding. NULL = expose all guest listening ports to host; empty array = expose none; entries = explicit map. Returns -ENOTSUP under passt. Note exposed ports are reached by host_port inside the guest too.
- `krun_add_vsock / krun_add_vsock_port / krun_add_vsock_port2 / krun_disable_implicit_vsock` `(ctx_id, tsi_features) / (ctx_id, port, unix_socket_path[, listen])` — Manage the vsock device and TSI hijack features (KRUN_TSI_HIJACK_INET/UNIX), plus host<->guest IPC over a host unix socket on a vsock port. By default an implicit vsock with TSI is created.
- `krun_set_rlimits` `(ctx_id, ["RESOURCE=CUR:MAX", ...])` — Set POSIX rlimits inside the guest before exec (e.g. NOFILE, NPROC). The in-guest resource-cap complement to vm_config. String-array format.
- `krun_set_exec / krun_set_workdir / krun_set_env` `(ctx_id, exec_path, argv[], envp[]) / (ctx_id, workdir) / (ctx_id, envp[])` — Set the workload command, working dir, and environment. envp NULL = inherit host env. Paths are relative to the configured root.
- `krun_setuid / krun_setgid` `(ctx_id, uid) / (ctx_id, gid)` — Drop to a uid/gid right before the microVM starts — lets the VMM open privileged host devices as root then run unprivileged. Directly relevant to single-user least-privilege.
- `krun_set_kernel / krun_set_firmware / krun_set_console_output / krun_get_shutdown_eventfd` `various` — Set external kernel/initramfs/cmdline, firmware, redirect console output to a file, and obtain an eventfd to request orderly guest shutdown (must be fetched before launch).
- `krun_check_nested_virt / krun_get_max_vcpus / krun_has_feature` `() / () / (feature)` — Capability probes — exactly the kind of preconditions our 'doctor' verb should run: does the host support nested virt, how many vCPUs max, was NET/BLK/GPU/TEE compiled into this libkrun build.
- `krun_start_enter` `(ctx_id)` — THE launch verb. Consumes the context, boots the microVM, takes over stdin/stdout, and on success NEVER returns — the VMM calls exit() with the workload's exit code when the VM shuts down. Returns only on pre-boot config error (-EINVAL).
- `krun_set_log_level / krun_init_log` `(level) / (target_fd, level, style, options)` — Configure library logging (OFF..TRACE) and log target fd / color style; KRUN_LOG_OPTION_NO_ENV blocks env-var overrides.

### Lifecycle model
Strictly EPHEMERAL and process-coupled: there is no daemon, no VM registry, no names, no persistence, no reaper. A microVM's lifetime IS the lifetime of the host process that called krun_start_enter — that call replaces the process (it exits() with the guest's exit code). Teardown is implicit: when the workload exits or the host process dies, the VM is gone. The only graceful-shutdown hook is the eventfd from krun_get_shutdown_eventfd (signal it to ask the guest to power off). Cleanup of host-side resources (mounts, the worktree, port bindings) is entirely the caller's responsibility — libkrun does none of it. This maps cleanly onto podman owning all lifecycle/labels above libkrun.

### Isolation & flags
- RESOURCE CAPS: krun_set_vm_config(num_vcpus, ram_mib) is the hard cpu/ram cap — this is the layer podman's krun.cpus/krun.ram_mib annotations target. Plus krun_set_rlimits for in-guest POSIX rlimits.
- FS PASSTHROUGH: krun_set_root/krun_add_virtiofs* expose host dirs by virtio-fs; read_only is ONLY available via krun_add_virtiofs3 (and add_disk read_only). Default krun_set_root is READ-WRITE. NO path containment is provided by the library.
- NETWORK DEFAULT: if no krun_add_net_* is called, TSI (Transparent Socket Impersonation) is auto-enabled — meaning the guest CAN reach the network by default. There is no in-library 'network none'; you get 'network none' only by disabling the implicit vsock (krun_disable_implicit_vsock) so no TSI device exists. This is why our --network none must come from podman, not libkrun.
- TSI port exposure default is permissive: krun_set_port_map(NULL) exposes ALL guest listening ports to the host; you must pass an empty array to expose none.
- PRIVILEGE: krun_setuid/krun_setgid let the VMM start as root then drop before boot.
- DISK FORMAT: krun_add_disk hard-restricts to raw; add_disk2/3 allow qcow2/vmdk but carry an explicit warning never to probe/format-detect untrusted images.
- Implicit-resource creation (console, vsock, init binary) is legacy; 2.0 API will remove it. krun_disable_implicit_* lets you opt out now.

### Standout features
- TSI (Transparent Socket Impersonation): networking with NO virtual interface — the VMM proxies AF_INET/INET6/UNIX sockets for the guest. Zero-config outbound+inbound connectivity, but it means VMM and guest share one network identity.
- Explicit, well-written SECURITY MODEL section in the README: it openly states guest and VMM are ONE security context and that virtio-fs + TSI are the two devices that leak host reach.
- Config-builder API shape (create_ctx -> many setters -> start_enter) — a clean separation of declaration vs launch.
- Capability-probe functions (krun_has_feature, krun_check_nested_virt, krun_get_max_vcpus) suitable for preflight checks.
- Built-in static init binary injected as /init.krun (can be disabled), so guests need no init of their own.
- Overlay virtual files/dirs (krun_fs_add_overlay_file/dir) for injecting host-memory-backed files like /etc/hostname without touching the host fs.
- Multiple variants (generic / SEV / TDX) with confidential-computing attestation as a build-time variant.

### Gotchas
- virtio-fs has NO sandbox: README states libkrun 'does not provide ANY protection against the guest accessing other directories in the same filesystem, or even other filesystems'. A guest can also exhaust inodes/disk. Host-side mount-namespace/quotas are REQUIRED. This is the security soft spot our brief flags — and it's why read-only mounts + minimal worktree exposure matter.
- Default krun_set_root is read-write; you only get read-only via the v3 virtiofs variant — easy to leave a share writable by accident.
- TSI is on BY DEFAULT (any time no net device is added), so 'no config' = 'guest has network'. Real network isolation requires disabling the implicit vsock, or for us, podman --network none.
- krun_set_port_map(NULL) silently exposes ALL guest listening ports to the host — fail-open, not fail-closed.
- krun_start_enter NEVER returns on success: it exit()s with the guest code. You cannot do host-side cleanup after it in the same process — cleanup must be via trap/parent process (mirrors our trap-based cleanup layer).
- Guest and VMM are the SAME security context: anything the VMM (i.e. the user running podman) can touch, the guest can potentially touch. Containment must come from the host OS layer (namespaces/SELinux/podman), not from libkrun.
- add_disk2/3 with non-raw formats can make the guest open arbitrary host files the image references — never format-probe an image a guest could have written.
- Exit codes 125/126/127 from init (env setup / cannot exec / not found) are distinct from the workload's own codes — worth preserving in our stable exit-code scheme.

### Fedora fit
Extremely Fedora-canon and the bedrock of our exact stack. It is the C in 'podman --runtime=krun': Fedora ships libkrun, libkrunfw, and crun's krun handler as packages. README's build/runtime requirements are written against Fedora package names (glibc-static, virglrenderer-devel, openssl-devel) and the example rootfs target uses podman to pull the Fedora container image. We never call this API directly — podman + crun-krun do — but it defines the real isolation surface beneath us.

### Lessons for our tool
- The FS-passthrough soft spot is real and library-level unfixable: libkrun explicitly disclaims any path jail. Our defense MUST be host-side — mount ONLY the per-sandbox worktree, mount everything else (or the worktree itself where feasible) read-only, and rely on podman's mount handling. Treat any rw host mount as full-filesystem-reach risk in the threat model.
- Map our resource caps onto the two real layers libkrun exposes: vm_config (cpu/ram, via podman krun.cpus/krun.ram_mib annotations) AND rlimits (NOFILE/NPROC) — caps on cpu/ram alone don't stop an inode/fd/proc blowup.
- Be fail-CLOSED where libkrun is fail-open: TSI-on-by-default and port_map(NULL)-exposes-all are the opposite of our --network none default. Confirms our decision to set --network none explicitly and never trust the engine's defaults.
- 'Guest and VMM are one security context' validates our SSH-agent-forwarding decision: never put keys/secrets in the guest, because guest reach ~= VMM (user) reach. Keep secrets on the host side of the socket.
- start_enter never returns on success => post-launch cleanup is impossible in-process. This is the library-level proof that our trap-on-INT/TERM/EXIT plus label-driven reap backstop (not in-process cleanup) is the correct architecture.
- Adopt the capability-probe pattern for our 'doctor' verb: emulate krun_check_nested_virt / krun_has_feature / krun_get_max_vcpus as host checks (KVM access, crun built with +LIBKRUN, crun-krun present) before we ever launch.
- Borrow the config-builder mental model and ONE-centralized-launch idea: libkrun funnels everything through a single ctx then a single start_enter — mirror this with our one centralized born-through function that stamps mandatory labels + safe-by-default flags.
- Preserve init's distinct exit codes (125/126/127 = setup/exec/not-found) separately from the workload exit code in our stable exit-code scheme, so callers can tell sandbox-setup failure from workload failure.
- read_only is a per-mount flag, not global — design our mount flag surface so read-only is the DEFAULT and writability is the explicit, narrow opt-in (only the worktree).

## libkrunfw (containers/libkrunfw)

**Tier:** documentation  |  **Language:** Makefile + Python (bin2cbundle.py) + C (generated bundle); kernel config files

**Purpose:** Bundles a Linux guest kernel into a dynamic library (libkrunfw.so) so libkrun can let the linker map the kernel sections into the VMM process and inject them straight into the guest with no extra processing. It is the actual kernel image our microVMs boot — pure build artifact, not a runtime CLI.


### Verb surface

- `make` `(no target) → builds default target 'all' = libkrunfw.so.<FULL_VERSION>; downloads linux-6.12.87 tarball, applies patches/, copies config-libkrunfw_<arch> to .config, runs olddefconfig, builds vmlinux, bin2cbundle→kernel.c, links the .so` — Build the generic Linux guest-kernel library
- `make SEV=1` `sets VARIANT=-sev, adds patches-tee/ patches, bundles qboot/sev/bios.bin + initrd/initrd.gz, uses config-libkrunfw-sev_x86_64 → libkrunfw-sev.so` — Build the AMD SEV confidential-computing kernel variant
- `make TDX=1` `sets VARIANT=-tdx, adds patches-tee/ patches, bundles qboot/tdx/bios.bin + initrd/initrd.gz, uses config-libkrunfw-tdx_x86_64 → libkrunfw-tdx.so` — Build the Intel TDX confidential-computing kernel variant
- `make OS=Windows` `CC=x86_64-w64-mingw32-gcc, VARIANT=-windows, config-libkrunfw-windows_x86_64 (Hyper-V enlightenments, 4K page align) → libkrunfw.dll` — Cross-compile the guest kernel library for the Windows build of libkrun
- `make install` `PREFIX (default /usr/local), DESTDIR; installs to $(PREFIX)/lib64 (Linux) and creates soname symlinks libkrunfw.so.<ABI=5> → libkrunfw.so` — Install the .so and ABI/soname symlinks
- `make clean` `none` — Remove kernel sources, generated kernel.c/qboot.c/initrd.c, and the built .so
- `make ARCH=<arm64|riscv> CROSS_COMPILE=<prefix>` `selects GUESTARCH aarch64/riscv64, uses config-libkrunfw_<arch>, builds Image instead of vmlinux` — Cross-build the guest kernel for another guest architecture
- `python3 bin2cbundle.py` `<input_file> <output_file> -t <vmlinux|Image|qboot|initrd> [--os Linux|Darwin|Windows]` — Convert a kernel/qboot/initrd binary into a C source blob (page-aligned: 64K default, 4K for Windows) to be compiled into the .so
- `./build_on_krunvm.sh` `env BUILDER=fedora|debian (default fedora); delegates to build_on_krunvm_<BUILDER>.sh` — macOS helper: spins up a krunvm Fedora/Debian VM to compile the kernel since you can't build a Linux kernel natively on macOS

### Lifecycle model
Build-time only — there is no runtime, no daemon, no persistent state, nothing to reap. The output is a single immutable shared library (libkrunfw.so.5.4.0, ABI_VERSION=5) that is dlopen'd by libkrun at VM-creation time. The only lifecycle concern is ABI/soname versioning: the soname carries ABI_VERSION (currently 5) and libkrun links against libkrunfw.so.5; a mismatched ABI between the installed libkrunfw and the installed libkrun is a hard failure. Pinned kernel: linux-6.12.87, version string 5.4.0.

### Isolation & flags
- Guest kernel is COMPILE-TIME fixed: the isolation surface is whatever this kernel config enables — our tool cannot change guest kernel features at runtime, only what podman/crun-krun/libkrun expose.
- CONFIG_NR_CPUS=16 in the standard x86_64 config (config-libkrunfw_x86_64); README's '8 CPU' limit actually reflects the SEV/TDX variants where CONFIG_NR_CPUS=8. So our krun.cpus annotation should stay well under the guest kernel's NR_CPUS ceiling (>16 CPUs would simply be ignored).
- Guest transports compiled in: CONFIG_VIRTIO_FS=y (virtio-fs is how host dirs are shared into the guest — NOT 9p; CONFIG_NET_9P is explicitly 'not set'), CONFIG_VIRTIO_VSOCKETS=y (vsock host<->guest), CONFIG_VIRTIO_BLK=y, CONFIG_VIRTIO_NET=y. Read-only/RW mount behavior is enforced by libkrun's virtiofs mapping, not the kernel.
- SEV variant: CONFIG_AMD_MEM_ENCRYPT=y, INTEL_TDX_GUEST off. TDX variant: CONFIG_INTEL_TDX_GUEST=y, AMD_MEM_ENCRYPT off. Confidential-compute variants additionally bundle a qboot BIOS (qboot/{sev,tdx}/bios.bin) and an initrd (initrd/initrd.gz) directly into the .so — the generic variant boots with neither.
- TEE patches (patches-tee/): restrict DMA when memory is encrypted, SEV AP reset vector, a driver to retrieve secrets from the kernel cmdline, avoid native_cpuid under SEV — i.e. the confidential variants change the early-boot/secret-injection path.

### Standout features
- Kernel-as-a-shared-library: the linker maps kernel ELF sections and libkrun injects those mappings into the guest with zero extra processing — elegant, but means the kernel is an opaque packaged blob to consumers.
- Multi-variant single-Makefile design: SEV=1 / TDX=1 / OS=Windows / ARCH= flags select patch sets, configs, BIOS, and initrd. Variant name is appended to the .so (libkrunfw-sev.so etc.) so variants coexist on disk.
- Confidential-compute variants embed qboot firmware + initrd into the library itself, so the generic and CC boot flows are materially different (firmware+initrd vs direct kernel).
- Reproducible-build hygiene: fixed KBUILD_BUILD_TIMESTAMP/USER/HOST and a pinned upstream kernel tarball + curated patch series.
- macOS/cross-build bootstrap via krunvm itself (build_on_krunvm.sh) — the project dogfoods the krun stack to build the krun stack.

### Gotchas
- The README '8 CPU' limitation is variant-specific: generic x86_64 is NR_CPUS=16, SEV/TDX are NR_CPUS=8. Don't quote '8' as a universal cap; verify against the actually-installed variant.
- ABI/soname coupling: libkrunfw and libkrun must agree on ABI_VERSION (5). On Fedora this is handled by package deps, but a hand-built or mismatched libkrunfw will break libkrun cryptically.
- Our tool boots whatever libkrunfw the Fedora package ships — it is the GENERIC variant, not SEV/TDX. SEV/TDX are irrelevant to our 'accident-not-adversary' threat model (they defend against the host/hypervisor, which we trust).
- Guest feature set is frozen at this kernel's config; if a workload needs a kernel feature/module not compiled in (e.g. 9p, or a niche fs), it simply isn't available in the guest. Plan around virtio-fs + virtio-vsock + virtio-net/blk only.
- build_on_krunvm_*.sh hard-codes a 2-CPU/2048MB krunvm builder and runs dnf builddep — purely a kernel-build helper, nothing to reuse for our runtime tool.

### Fedora fit
Highly Fedora-canon. Maintained under github.com/containers (same org as podman/crun/libkrun), explicitly documents Fedora package names (python3-pyelftools), provides a Fedora-based krunvm builder, and is packaged in Fedora as the libkrunfw RPM that libkrun depends on. Our locked engine (podman --runtime=krun over official Fedora packages) consumes exactly this artifact — we should never build it ourselves.

### Lessons for our tool
- This repo is NOT a verb-surface model for our CLI — its only 'verbs' are build targets (make / make install / make clean). It informs WHAT our microVM boots, not how we name commands. Look elsewhere (krunvm, podman) for verb naming.
- Treat libkrunfw as an opaque, package-managed dependency. Our 'doctor' verb-zero check should verify the libkrunfw package/.so is present and ABI-compatible with the installed libkrun (the krun runtime won't start without it), alongside the KVM-access and crun+LIBKRUN checks.
- Design isolation around the guest kernel's real capabilities: host-dir sharing is virtio-fs (read-only vs RW enforced by the mount config we pass to libkrun), host<->guest control is virtio-vsock, networking is virtio-net. Don't assume 9p or arbitrary kernel modules exist.
- Respect the guest NR_CPUS ceiling: our conservative krun.cpus annotation is well within range, but document that requesting more vCPUs than the guest kernel's NR_CPUS silently caps — never error.
- Skip SEV/TDX entirely. They target a host-distrust threat model; ours is accident-containment with a trusted host, so the generic Fedora-shipped libkrunfw is exactly right and we should not surface any confidential-compute knobs.
- Version-pinning discipline is worth imitating: like libkrunfw pins kernel+ABI, our tool should record the libkrun/crun-krun/podman versions it was validated against (e.g. in doctor output) so guest-behavior surprises are diagnosable.

## krunvm (github.com/containers/krunvm, Red Hat / Sergio Lopez)

**Tier:** documentation  |  **Language:** Rust (clap derive arg parser; calls out to buildah + libkrun FFI)

**Purpose:** A CLI utility to create and run microVMs directly from OCI images using libkrun + buildah, with zero disk-image maintenance and zero network config. Each microVM is a buildah-mounted OCI rootfs booted via libkrun's krun_start_enter().


### Verb surface

- `create` `<IMAGE> [--name NAME] [--cpus N] [--mem MiB] [--dns IP] [-w/--workdir PATH] [-v/--volume host:guest ...] [--port host:guest ...] [--x86 (macOS only)]` — Pulls/uses an OCI image via `buildah from`, mounts its rootfs, writes .krun_config.json + a fixed /etc/resolv.conf, unmounts, and PERSISTS the VM definition (name->VmConfig) into the krunvm confy config file. Prints 'microVM created with name: <name>'. Name defaults to the buildah container id if --name omitted. Rejects duplicate names. This is a registration/template step, not a boot.
- `start` `<NAME> [COMMAND] [ARGS...] [--cpus N (TODO/no-op)] [--mem MiB (TODO/no-op)] [--env key=value ...]` — Boots the named persisted microVM: re-mounts the buildah rootfs, sets cpus/mem from stored config, maps volumes (bind-mount on Linux, virtiofs on macOS) and ports, sets workdir/env, takes an flock lockfile (.krunvm.lock) to prevent concurrent boots of the same VM, then krun_start_enter() — the process REPLACES into the VM and blocks until exit. If COMMAND omitted it runs the image's default. Injects HOSTNAME=<name> and HOME=/root.
- `list` `[-d/--debug (TODO/no-op)]` — Prints all persisted microVMs from config: name, CPUs, RAM, DNS, buildah container id, workdir, mapped volumes, mapped ports. 'No microVMs found' if empty.
- `inspect` `<NAME>` — Runs `buildah inspect` on the VM's underlying buildah container and prints raw output. A thin passthrough to buildah, not a krunvm-native format.
- `delete` `<NAME>` — Removes the VM from config, unmounts the buildah container, and `buildah rm`s it (remove_container). Permanent teardown of the template+rootfs.
- `changevm` `<NAME> [--new-name NAME] [--cpus N] [--mem MiB] [-w/--workdir PATH] [--remove-volumes] [-v/--volume ...] [--remove-ports] [--port ...]` — Mutates a persisted VM's stored config in place (rename, resource caps, workdir, replace/clear volume & port maps), re-saves config, prints the updated VM. Note: --volume is IGNORED if --remove-volumes also given (same for ports) — a known TODO.
- `config` `[--cpus N] [--mem MiB] [--dns IP]` — Reads/writes GLOBAL defaults (default_cpus, default_mem, default_dns) applied to future `create`s. With no args, just prints current global defaults.

### Lifecycle model
PERSISTENT-by-default, the opposite of our tool. `create` registers a named VM into a confy TOML/YAML config file (the state file: KrunvmConfig.vmconfig_map) and leaves a buildah container + mounted/unmounted rootfs on disk. `start` boots it but does NOT delete it on exit; the same template can be started repeatedly. Teardown is fully MANUAL via `delete` (unmount + buildah rm + drop from config). There is NO --rm, NO ephemeral mode, NO trap-based cleanup, NO reaper/age logic, NO automatic GC. The only runtime safety is an flock lockfile per-VM preventing two concurrent boots of the same VM. State lives in two places that can drift: the confy config file AND buildah's container store.

### Isolation & flags
- Resources: --cpus and --mem set at create time, stored, enforced at boot via krun_set_vm_config(cpus as u8, mem). HARD CAPS: cpus<=8, mem<=16384 MiB enforced in changevm/config (NOT enforced in create's path beyond defaults). Global defaults: 2 vCPUs, 1024 MiB.
- Network: 'zero network configuration' — libkrun provides user-mode/passt-style networking automatically; there is NO --network flag and NO way to disable networking. DNS is the only network knob (--dns, default 1.1.1.1, written into guest /etc/resolv.conf). This is the OPPOSITE of our --network none default.
- Volumes: -v/--volume host_path:guest_path, repeatable, stored in config. On Linux mapped via libc bind-mount (MS_BIND|MS_REC) into the rootfs; on macOS via krun_add_virtiofs + a generated /.krunvm-mount.sh. NO read-only option — all mounts are read-write bind mounts.
- Ports: --port host_port:guest_port, repeatable, exposed via krun_set_port_map.
- No annotations/labels concept at all — krunvm predates and bypasses podman/OCI annotations; it talks to buildah+libkrun directly, so there is no label-driven anything.
- RLIMIT_NOFILE is raised to max at boot (set_rlimits) so the VM gets enough fds.
- Linux requires running inside `buildah unshare` (a rootless user-namespace) — checked via check_unshare(): refuses to run as non-root unless BUILDAH_ISOLATION is set.

### Standout features
- Zero disk-image maintenance: boots an OCI image's rootfs directly (buildah mount) — no qcow2/raw image to build or resize. The 'container as VM template' model.
- Process-replacement execution model: `start` calls krun_start_enter() which turns the current process INTO the VM; stdio is the VM console, exit code flows back. No daemon, no detached VM — very script-friendly for a single foreground workload.
- Per-VM flock lockfile (.krunvm.lock in rootfs) is a clean, stateless concurrency guard against double-booting the same VM.
- Clear separation of global defaults (`config`) vs per-VM overrides (`create`/`changevm`) vs runtime (`start`).
- macOS Rosetta/x86-on-aarch64 support and case-sensitive APFS volume bootstrapping (not relevant to us, Fedora-only).

### Gotchas
- Two sources of truth: the confy config file and buildah's container store. If they drift (e.g. buildah container removed externally), krunvm has no reconciliation — exactly the bug our 'podman is the only source of truth, no state file' decision avoids.
- --cpus/--mem on the `start` subcommand are declared but are dead TODOs (no-op); resources are only settable at create/changevm. Likewise `list -d` and the top-level `-v` verbosity are unimplemented TODOs. Lesson: don't ship flags you don't wire up.
- changevm silently ignores --volume when --remove-volumes is also passed (and ports likewise) — confusing UX, flagged as TODO in source.
- No --rm / no ephemeral path: every `create` leaves persistent state you must `delete`, so an agent loop would accumulate buildah containers + config entries indefinitely. No reaper.
- All volume mounts are read-write bind mounts with no read-only option — bad for our blast-radius/accident threat model.
- Linux: hard requirement to be inside `buildah unshare`; running plainly will exit -1 with a message. Adds a wrapping step.
- Networking cannot be turned off; default DNS is 1.1.1.1 baked into guest resolv.conf.
- Uses exit code -1 everywhere on error (becomes 255), with no distinct/stable exit codes — poor for scripting.

### Fedora fit
COPR-only on Fedora (slp/libkrunfw, slp/libkrun, slp/krunvm), NOT in official Fedora repos. We explicitly do NOT use krunvm. It also bypasses podman/crun entirely (talks buildah + libkrun directly), so it is architecturally OFF our path (our engine is `podman run --runtime=krun` over official packages). Value here is purely as verb-shape prior art, not as a dependency.

### Lessons for our tool
- Verb vocabulary to borrow/contrast: krunvm splits create (register template) from start (boot). Our ephemeral-by-default tool can COLLAPSE these into one verb (e.g. `run`) that creates+boots+cleans in one shot, reserving a separate persist opt-in. Keep `list`, `delete`, and a buildah-passthrough-style `inspect` as recognizable names. `config` for global defaults and `changevm` for per-instance mutation are good names if we ever need them — though our no-state-file rule means we likely don't.
- Their persistent confy state file is the anti-pattern our design already rejects: it drifts from the real container store. Reaffirms 'podman labels are the only source of truth, no state file.'
- Resource cap UX: store cpus/mem as integers with hard ceilings (they cap cpus<=8, mem<=16384). We should similarly clamp/validate krun.cpus / krun.ram_mib annotations and pick conservative defaults (they default 2 CPU / 1024 MiB — reasonable starting points for us).
- Concurrency: the per-VM flock lockfile is a cheap, stateless double-boot guard. If we ever need to prevent two sandboxes claiming the same worktree/port, an flock keyed on the worktree path is a clean pattern that needs no daemon.
- Don't ship dead flags: krunvm has multiple declared-but-unimplemented options (start --cpus/--mem, list -d, -v). Our one disciplined bash script should only expose flags it actually wires to podman.
- Network default is the key divergence: krunvm forces user-mode networking on with no off switch. We invert this — --network none is the SAFE default, opt-in to publish. Good confirmation our default is the careful one.
- Mounts: krunvm offers only read-write bind mounts and no :ro. For our accident-not-adversary model, default mounts to read-only (:ro) and require an explicit flag for read-write — directly addresses a krunvm gap.
- Stable exit codes: krunvm returns -1 (255) for every error. We should define distinct, documented exit codes (precondition fail vs not-found vs runtime fail) for our scriptable/--json contract.
- Single-foreground-process model (krun_start_enter replaces the process) maps perfectly onto `podman run --rm` in the foreground with our trap-on-INT/TERM/EXIT cleanup — confirms the foreground-blocking design and the value of a label-driven reap backstop for the crash case krunvm has no answer for.
- Precondition check (their check_unshare / macOS volume bootstrap) validates our plan for a `doctor` verb-zero: detect KVM access, crun+LIBKRUN, crun-krun before doing work, with a clear actionable message rather than a deep failure.

## crun-vm

**Tier:** documentation  |  **Language:** Rust (binary is a thin OCI-runtime shim that ultimately drives libvirt/QEMU via crun/crun-krun + embedded bash scripts)

**Purpose:** crun-vm is itself an OCI Runtime (a drop-in alternative to crun, set via --runtime crun-vm) that lets Podman/Docker/Kubernetes boot full QEMU-compatible VM images (containerdisks, raw qcow2 dirs, or bootc bootable containers) as if they were containers. It is the "full VM image" sibling to the krun microVM path: same engine verbs, but the workload is a complete VM booting via libvirt+QEMU rather than a microVM.


### Verb surface

- `(none of its own — it is an OCI runtime, not a CLI tool)` `--runtime crun-vm` — crun-vm exposes NO user-facing verbs. The user-facing verbs are ALWAYS the engine's: podman/docker create|run|exec|attach|stop|start|rm and kubectl. crun-vm is selected via the engine's --runtime crun-vm flag (or Quadlet GlobalArgs=--runtime crun-vm). This is the central architectural fact: verb surface == podman's.
- `create` `<oci create args> (called by engine)` — OCI runtime verb that crun-vm INTERCEPTS and implements itself (src/commands/create). Rewrites the OCI config.json into a libvirt domain that boots the VM image, handles --rootfs, bootc->VM-image conversion+caching, first-boot (cloud-init/Ignition/password), virtiofs mounts, blockdevs, port forwarding setup.
- `delete` `<oci delete args> (called by engine)` — OCI runtime verb crun-vm intercepts (src/commands/delete.rs) to tear down the VM/domain and crun-vm's auxiliary state when the engine removes the container.
- `exec` `<oci exec args> (called by engine on podman/docker exec)` — OCI runtime verb crun-vm intercepts (src/commands/exec.rs). Instead of nsenter-ing the container, it SSHes into the running VM. Supports crun-vm-specific prefix opts --as <user> (default root), --timeout, and --container (enter the hypervisor container instead of the VM).
- `start / state / kill` `<oci args> (called by engine)` — NOT reimplemented — crun-vm passes these straight through to crun (see lib.rs match arm calling crun(&raw_args)).
- `run/list/ps/pause/resume/update/checkpoint/events/features/spec` `n/a` — Explicitly NOT supported as runtime verbs — bail!("Unknown command"). (Engine-level podman run still works; it decomposes into create+start which crun-vm handles.)

### Lifecycle model
Lifecycle is entirely the ENGINE's: the VM lives exactly as long as the container object. Ephemeral vs persistent is controlled by standard podman flags (--rm for teardown, -d/--detach for background) — crun-vm adds NO reaper/age logic of its own. Teardown: `podman stop --latest` / `podman rm`; the intercepted OCI `delete` cleans up the libvirt domain. Detach/reattach: ctrl-p ctrl-q then `podman attach --latest`; abort VM with ctrl-]. DISK persistence is orthogonal and opt-in: by default writes are throwaway; the crun-vm-specific `--persistent` flag (only with --rootfs) writes back into the user's image file (with an explicit data-corruption warning). For long-lived service VMs, the recommended persistence model is Podman Quadlet/systemd units (generated via podlet), i.e. systemd owns the lifecycle, not crun-vm.

### Isolation & flags
- Runtime selection: --runtime crun-vm (engine flag). All crun-vm-specific opts go AFTER the image ref, in the slot where a container command would normally be (since VMs have no 'command'); Docker needs an extra empty "" arg if the image has no entrypoint.
- Network: NO crun-vm-specific network flag. Uses standard engine networking; port exposure is plain podman -p/--publish host:guest (TCP/UDP), implemented inside the VM via passt. There is no '--network none' equivalent baked in — isolation is whatever the engine default is.
- CPU: NO crun-vm annotation. It DERIVES vCPU count from the standard OCI cgroup cpu quota/period (i.e. podman --cpus). Falls back to num_cpus::get() (ALL host CPUs) if unset. cpuset is honored as libvirt vcpu cpuset. (Contrast krun.cpus annotation.)
- Memory: NO crun-vm annotation. DERIVES VM RAM from the standard OCI memory limit (podman --memory). Default if unset: 2 GiB (2^31 bytes). (Contrast krun.ram_mib annotation.)
- Mounts: standard -v/--mount/--volume bind mounts are surfaced INTO the VM as virtiofs (auto-mounted at destination if cloud-init/Ignition present, else manual `mount -t virtiofs virtiofs-<index>`). Regular files appear as block devices. SELinux :z relabel works.
- Block devices: standard --device host:guest (assumed raw), OR crun-vm-specific --blockdev source=,target=,format= (absolute paths; supports qcow2/raw/any QEMU format; file or device).
- First-boot config: --password <plaintext> (cloud-init default user), --cloud-init <abs dir>, --ignition <abs file>. authorized_keys auto-wired for root + default/core user so exec-SSH works.
- SSH keys: by default the HOST user's existing ssh key pair is used to exec into the VM; --random-ssh-key-pair forces a fresh ephemeral pair instead.
- Emulation: --emulated runs QEMU in software (no KVM); NOT allowed for bootc images. --bootc-disk-size sets generated bootc VM disk size (default 2x image).
- Debug/introspection: --print-libvirt-xml, --merge-libvirt-xml <abs>, --print-config-json (dump modified OCI config and exit).
- All crun-vm path options MUST be absolute (enforced; because docker/k8s don't preserve cwd).

### Standout features
- Reuses the entire engine UX: containers and full VMs managed with ONE toolset (podman/docker/kubectl/systemd-quadlet). No new verbs to learn.
- Three image input modes: containerdisk (image with a lone disk file under / or /disk), --rootfs dir with a lone qcow2, and bootc bootable container (converted to a VM image then cached as a containerdisk for fast reboots).
- exec is SSH-into-the-VM, transparently, with --as <user>; --container escape hatch to reach the hypervisor container.
- Resource caps come for free from standard OCI/podman --cpus/--memory rather than bespoke annotations.
- Quadlet/systemd integration story for persistent service VMs (podlet generates the unit).
- Built on the SAME lower stack our project uses: depends on crun + crun-krun, plus libvirtd/virtqemud, qemu-img, passt, genisoimage.

### Gotchas
- crun-vm-specific flags must come AFTER the image ref (in the command slot), not among engine opts — easy to misplace; and podman-exec often needs `--` before crun-vm opts (e.g. `podman exec --latest -- --as fedora`).
- No command is allowed in the run slot for VMs; with Docker you must pass an empty "" to satisfy syntax if the image has no entrypoint.
- Default RAM is 2 GiB and default vCPU = ALL host CPUs when --memory/--cpus are unset — silently generous; you must set podman --memory/--cpus to constrain a VM.
- --persistent only works with --rootfs and can CORRUPT the image if the file is used by two VMs at once; conflicts with a read-only root.
- --emulated cannot be used with bootc images.
- Heavy runtime dependency surface (libvirtd/virtqemud, qemu-system-*, passt, genisoimage, qemu-img) — far more than a microVM path needs; boot is slower and the attack/blast surface is larger.
- bootc images do a one-time convert-and-cache step that quietly writes a generated containerdisk into host container storage.
- Files bind-mounted in appear as block devices (not files) inside the VM — surprising; subject to change.

### Fedora fit
Strongly Fedora-canon: installed via `dnf install crun-vm`, lives under the containers org (github.com/containers/crun-vm), ships an fmf/.packit.yaml (Fedora CI/packaging), and is registered as a Podman runtime in containers.conf. It deliberately sits on the official Fedora container stack (crun, crun-krun, libkrun-adjacent, podman). It is canon for the FULL-VM use case, whereas our project targets the microVM (krun) use case on the same blessed stack.

### Lessons for our tool
- Strongest contrast lesson: crun-vm proves you can ship a VM-execution tool that invents ZERO verbs of its own — it rides podman's verb surface entirely via --runtime. Our tool deliberately goes the OTHER way (a single wrapper script with our own flag-preceded verbs) so we own UX, labeling, and cleanup; crun-vm validates that podman-as-source-of-truth is viable, reinforcing our no-daemon/no-state-file stance.
- Borrow the exec model: 'enter the workload' should be SSH-into-the-guest with an explicit user (their --as <user>, default root). Pair this with our SSH-agent-forwarding decision (keys never enter the guest). crun-vm's default of reusing the host's key pair is the same spirit.
- Resource caps: crun-vm derives CPU/RAM from STANDARD podman --cpus/--memory (OCI cgroup quota), NOT custom annotations. We chose krun.cpus/krun.ram_mib annotations — fine, but note the alternative is engine-native flags. Either way, COPY their lesson of NOT leaving it unset: their unset default (2 GiB + ALL host CPUs) is exactly the unsafe-default we must avoid; our centralized birth function must always stamp conservative caps.
- Persistence as explicit opt-in: their --persistent flag (with a loud data-corruption warning and a conflict check vs read-only root) mirrors our ephemeral-by-default + opt-in-persistence-verb decision. Steal the loud warning and the guard.
- Path discipline: they force ALL path options to be absolute because cwd is unreliable under the engine. Our bash script should likewise resolve/require absolute paths before handing anything to podman, to avoid cwd-dependent surprises.
- Quadlet/systemd is THE persistence/service story on this stack — if a user wants a long-lived sandbox, point them at a Quadlet unit rather than growing our own daemon (keeps us no-daemon).
- Dependency weight is a selling point for the krun path: crun-vm needs libvirtd, qemu-system-*, passt, genisoimage, qemu-img. Our krun microVM avoids most of that — smaller blast radius, faster boot, simpler doctor check. Our 'doctor' verb-zero should NOT require libvirt/qemu; just KVM + crun(+LIBKRUN) + crun-krun.
- Networking: crun-vm has no built-in isolation default — it inherits engine networking and only adds -p forwarding. This is the gap our project closes by defaulting --network none and publishing predictably to localhost; do NOT copy their permissive default.
- Flag placement footgun: crun-vm's 'opts after the image, sometimes needing --' is a real UX wart. Our single wrapper with flag-preceded args avoids this entirely — a concrete reason our form factor is better for a single-user tool.
- They cleanly separate 'enter the VM' vs 'enter the hypervisor container' (--container). If we ever expose a debug path into the krun VM vs the podman container, give it a distinct, explicit flag rather than overloading exec.

## podman (containers/podman) v5.6.0

**Tier:** documentation  |  **Language:** Go (CLI); docs are Markdown with @@option include templating)

**Purpose:** Daemonless OCI container/pod engine. `podman run` starts a process in a new container with its own filesystem, network, and PID tree; it is the exact CLI surface our tool wraps (podman run --runtime=krun ...). It is the only source of truth for container metadata via labels/annotations.


### Verb surface

- `podman run` `[options] IMAGE [command [arg...]]` — Create + start a container in one step (alias: podman container run). The verb our tool wraps. Pulls image if absent, then runs.
- `podman create` `[options] IMAGE` — Create a container WITHOUT starting it (separate create/start from run). Same flag set as run minus run-only flags.
- `podman start` `[opts] CONTAINER` — Start a previously-created/stopped container. Pairs with create for two-phase lifecycle.
- `podman stop` `[--time N] CONTAINER` — Send SIGTERM then SIGKILL after timeout. Graceful stop.
- `podman kill` `[--signal SIG] CONTAINER` — Send a signal (default SIGKILL) to the main process. Hard kill, no grace.
- `podman rm` `[--force/-f] [--volumes/-v] [--all/-a] [--depend] [--time N] [--filter/-f] CONTAINER` — Remove a container. -f stops+removes a running one; -v also removes anonymous volumes. Supports --filter (label=, status=, etc). This is the reaper primitive.
- `podman ps` `[-a] [-q] [--filter/-f KEY=VAL] [--format]` — List containers. --filter label=k=v / status=exited / until=DUR / annotation=k=v drives label-based discovery. -q prints IDs, --format Go-template for machine-readable output.
- `podman exec` `[-it] CONTAINER cmd` — Run an additional command in a running container.
- `podman inspect` `[--format] CONTAINER` — Dump full JSON config incl Labels, Annotations, OCIRuntime. Source of truth for our metadata.
- `podman container prune` `[--filter until=]` — Bulk-remove all stopped containers; supports until/label filters. A coarse reaper.
- `podman container exists` `CONTAINER` — Exit 0/1 existence check; good for idempotency guards in scripts.
- `podman container cleanup` `CONTAINER` — Run the post-stop cleanup (unmount, network teardown) that normally runs automatically; internal lifecycle hook.
- `podman events` `[--filter] [--since]` — Stream lifecycle events (create/start/died/remove). Could feed an event-driven reaper instead of polling.
- `podman quadlet install` `FILE.container` — Install a Quadlet unit for current user (new in 5.6, rootless). Not the model for us (systemd-managed, persistent).
- `podman quadlet list/print/rm` `` — Manage installed Quadlets. Quadlet = systemd-unit-driven containers; persistent, declarative — opposite of our ephemeral --rm model.
- `podman generate systemd` `CONTAINER` — DEPRECATED in favor of Quadlet; emits a .service from a running container.

### Lifecycle model
Two distinct models. (1) Imperative ephemeral: `podman run --rm` auto-removes the container AND its anonymous unnamed volumes when it exits (default --rm is false). `--rmi` removes the image too (implies --rm). No daemon: there is no background reaper — a crashed/SIGKILLed `podman run` can orphan a still-registered (Exited) container, which is exactly why a label-driven backstop sweep is needed. The autoremove behavior is recorded as the annotation io.podman.annotations.autoremove and is queryable via `podman ps --filter annotation=io.podman.annotations.autoremove=false`. (2) Declarative persistent: Quadlet (.container/.volume/.network unit files) + systemd manages lifecycle/restart — explicitly NOT ephemeral. Cleanup primitives available to a reaper: `podman ps -a -q --filter label=...`, `podman rm -f -v <id>`, `podman container prune --filter until=`. Filter keys relevant to a sweep: label (EXCLUSIVE matching), status=exited, until=DateTime/duration, exited=INT (exit code), since/before.

### Isolation & flags
- --runtime=PATH|NAME: select OCI runtime (this is a GLOBAL podman flag, `podman --runtime crun-krun run ...`, NOT a run subflag; root podman.1 default on cgroupv2 is crun). --runtime-flag passes runtime global flags (strip leading --).
- --network=MODE (alias --net): none|bridge|private|host|pasta|container:id|ns:path|<name>. 'none' = namespace but NO interfaces/connectivity (our safe default). 'host' is flagged in docs as a security vulnerability. rootless default is pasta; rootful default is bridge.
- --annotation=key=value (repeatable): OCI annotations. THIS is the channel libkrun/crun-krun reads (e.g. run.oci.handler, krun.cpus, krun.ram_mib) — note these krun-specific keys are NOT documented in podman docs; podman just passes annotations through to the runtime. Filterable via ps --filter annotation=.
- --label/-l key=value (repeatable): container metadata. ps --filter label=k=v is EXCLUSIVE (multiple labels AND together). label!=k filters NOT-having. This is our state store (no daemon/state file).
- --rm: auto-remove container + anonymous volumes on exit (default false). --rmi also removes image.
- --read-only: root fs read-only (default writable). --read-only-tmpfs (default TRUE): when --read-only, mounts rw tmpfs on /dev,/dev/shm,/run,/tmp,/var/tmp; set false for a totally read-only container.
- -v/--volume [src:]dst[:OPTS]: bind mount; OPTS include ro|rw (default rw), z|Z (SELinux relabel), O (overlay), U (chown), [no]exec/[no]dev/[no]suid, idmap, r-bind/rslave etc. Named volumes are NOT removed by --rm; anonymous ones ARE.
- --mount type=bind|tmpfs|volume|image|ramfs|devpts|glob,src=,dst=,...: more explicit modern mount form with ro/relabel suboptions.
- --memory/-m=N[bkmg]: hard memory cap (0=unlimited). NOTE: not supported on cgroups v1 rootless. (krun also honors krun.ram_mib annotation.)
- --cpus=N: CPU count cap (0.0=unlimited; shorthand for cpu-period/quota; cannot combine with them). Not supported on cgroups v1 rootless. (krun honors krun.cpus annotation.)
- --security-opt: no-new-privileges, seccomp=profile.json|unconfined, apparmor=, label=disable/nested, mask=/unmask=, proc-opts=. Default masked /proc paths and read-only /proc,/sys subpaths are listed in docs.
- --cap-add/--cap-drop, --userns (host/keep-id/auto/private etc), --cgroupns, --publish/-p [ip:][hostPort:]containerPort[/proto] (publish only works with bridge/pasta, NOT host/none), --replace (replace existing same-name container), --name, --shm-size.

### Standout features
- Daemonless + rootless: no central broker; the container store IS the state. Labels/annotations are first-class query keys via ps --filter.
- --runtime is pluggable and is the entire integration point for krun/libkrun (microVM) — podman is unaware it's a VM; it just hands an OCI bundle + annotations to the runtime.
- --rm couples container teardown with anonymous-volume teardown in one flag.
- Rich --filter grammar (label exclusive-AND, status, until/since/before, exited code, annotation, regex on name/id) — enough to build a reaper with zero extra state.
- --format Go-templates everywhere (ps/inspect) give stable machine-readable output without parsing human tables.
- --replace gives idempotent re-create by name.
- autoremove status is itself persisted as an annotation and is queryable.
- Quadlet is the sanctioned persistence path (systemd units) if/when we want managed long-lived sandboxes.

### Gotchas
- --rm only removes ANONYMOUS volumes; named volumes survive --rm (need `podman rm -v` / `podman volume rm`). A worktree bind-mount is a host path, not a volume, so it is never auto-deleted — good, but the container record can still orphan.
- No daemon means no automatic reaper: a SIGKILLed `podman run` leaves an Exited container row. The label-sweep backstop is mandatory, not optional.
- --memory and --cpus are silently unsupported on cgroups v1 rootless; on krun microVMs prefer the krun.ram_mib/krun.cpus annotations (which podman docs do NOT mention — they are runtime-private).
- --runtime is a GLOBAL flag (podman --runtime ... run), easy to mis-place after the subcommand.
- --network none means port --publish does nothing (publish requires bridge/pasta); plan localhost publishing only when you actually want connectivity.
- --read-only-tmpfs defaults to TRUE, so a --read-only container still has writable /tmp,/run,/var/tmp unless you set it false.
- SELinux: bind mounts on Fedora often need :z/:Z relabel or they'll be denied; :Z is private-label and can clobber shared host dirs.
- --host network shares abstract unix sockets + localhost TCP with host (docs explicitly call it a security vulnerability).
- ps --filter label is EXCLUSIVE (AND) while other keys are also exclusive across different keys — get the semantics right when composing a sweep query.
- Quadlet install/list/print/rm are NOT available on the remote podman client in 5.6.

### Fedora fit
Maximally Fedora-canon. Podman is the Fedora-default container engine, shipped and maintained by Red Hat; crun is the default OCI runtime on cgroups v2 (Fedora's default). crun-krun + libkrun + libkrunfw are official Fedora packages. The whole engine stack our project fixes on is first-party Fedora, no third-party binaries, no daemon, rootless by default.

### Lessons for our tool
- Wrap `podman run` directly (the verb our script proxies); offer create+start only if we ever need two-phase. Put --runtime BEFORE the run subcommand (it's a global flag): `podman --runtime crun-krun run ...`.
- Use LABELS, not a state file, as the single source of truth — podman is daemonless and ps --filter label=k=v (exclusive AND) lets us list/reap by our mandatory label set (sandbox.managed-by/.created/.id/.worktree). Confirm the no-state-file decision is well-founded.
- Pass krun caps (krun.cpus, krun.ram_mib) and run.oci.handler via --annotation, NOT --memory/--cpus — those cgroup flags are unreliable on rootless cgroups v1 and are not the krun-native knob. Annotations are also filterable (ps --filter annotation=).
- Three-layer cleanup maps cleanly onto podman: layer1 = --rm (auto-removes container + anonymous volumes on clean exit); layer2 = bash trap calling `podman rm -f -v <id>`; layer3 = reaper sweep `podman ps -a -q --filter label=sandbox.managed-by=<us> --filter status=exited` then `podman rm -f -v`. Consider `podman container prune --filter until=` / a .created-label age check for the age backstop.
- Safe-by-default isolation flag set to centralize: --network none, --read-only with --read-only-tmpfs (note it defaults true so /tmp stays writable; set false for fully-RO), --security-opt no-new-privileges, conservative --cap-drop. The worktree is bound rw with :Z (SELinux) — everything else ro.
- Worktree is a host bind-mount (-v /path:/work:Z), NOT a volume, so --rm will never delete it — exactly the orphan-safety we want; but remember to add :z/:Z for SELinux on Fedora or mounts get denied.
- For machine-readable output, lean on `podman ps --format '{{.ID}} {{.Labels}}'` and `podman inspect --format` Go-templates rather than parsing human tables — gives us our --json/stable-columns mode for free.
- Doctor verb-zero can use `podman info`, `podman --runtime crun-krun run ... true` smoke test, and check crun feature flags / KVM access; `podman container exists` style exit-code checks model our stable exit codes.
- Publishing to localhost only works on bridge/pasta networks — our 'publish predictably to localhost' path must NOT use --network none for that sandbox, and --publish + --network none is a silent no-op to guard against.
- --replace gives idempotent recreate-by-name if we ever key sandboxes by name; otherwise random names + label.id keep concurrency simpler.
- Quadlet/systemd is the sanctioned graduation path for persistence — but it is declarative+systemd-managed (opposite of ephemeral). Keep our explicit persistence opt-in as a label stamp + skip-of---rm, NOT a Quadlet, until the graduation signal hits.
- Avoid --network host (docs call it a security vulnerability) and avoid relying on cgroup-v1 rootless resource caps — both are footguns for our accident-not-adversary threat model.

## superradcompany/microsandbox (local clone at /var/home/tom/mecattaf/dotfiles/skills/microvm/references/inspiration/microsandbox)

**Tier:** inspiration  |  **Language:** Rust (CLI + core SDK crate; plus Python/Node-TS/Go SDK bindings). In-guest agent `agentd` is a separate Rust binary.

**Purpose:** A full microVM sandboxing platform (libkrun-based) that gives AI agents and developers Docker-like control over disposable VMs to run untrusted/agent-generated code. Ships a `msb` CLI, a multi-language SDK, an MCP server, and Agent Skills — far broader than a single-user bash tool.


### Verb surface

- `msb run` `<IMAGE|--snapshot PATH_OR_NAME> [-d/--detach] [-t/--tty] [--timeout DUR] [--rlimit nofile=N] [--detach-keys] [+ all SandboxOpts] -- <cmd...>` — Create a sandbox from an image (or snapshot) and run a command in it. UNNAMED sandboxes are ephemeral: auto-removed after the command exits. NAMED sandboxes are reused if they already exist (unless --replace). This is the primary one-shot verb.
- `msb create` `<IMAGE> [SandboxOpts]` — Create a sandbox and boot it in the BACKGROUND (persistent/detached), ready for later `exec`. Does not run a foreground command.
- `msb start` `<NAME...> [-q]` — Start one or more previously-created (stopped) sandboxes, detached.
- `msb stop` `<NAME...> [-f/--force] [-t/--timeout SECS] [-q]` — Stop running sandboxes; graceful SIGTERM then SIGKILL after timeout. --force kills immediately (may lose unsynced writes).
- `msb list (alias: ls)` `[flags]` — List all sandboxes (the DB-tracked inventory).
- `msb status (alias: ps)` `[NAME] [--format json]` — Show sandbox status. Supports `--format json` for machine-readable single or array output.
- `msb metrics` `<NAME>` — Show live metrics (CPU/mem) for a running sandbox; streams.
- `msb remove (alias: rm)` `<NAME...> [-f/--force] [-q]` — Remove sandboxes from inventory; --force stops-then-removes a running one.
- `msb exec` `<NAME> [-e ENV] [-w WORKDIR] [-u USER] [-t/--tty] [--timeout] [--rlimit] -- <cmd...>` — Run a command inside an already-running (or auto-started) sandbox. No command in interactive mode = attach to default shell. Auto-starts the sandbox if stopped (resolve_and_start).
- `msb copy (alias: cp)` `<src> <dst> (host<->sandbox paths)` — Copy files between host and a sandbox.
- `msb logs` `<NAME> [flags]` — Show captured stdout/stderr output from a sandbox (post-mortem via runtime.log).
- `msb image` `subcommands: pull | list(ls) | inspect | remove(rm)` — Manage OCI images cached locally.
- `msb pull` `<IMAGE>` — Top-level alias to download an image from a registry (same as `image pull`).
- `msb registry` `subcommands: login | logout | list(ls)` — Manage registry credentials.
- `msb ssh` `[NAME] | connect <NAME> | serve <NAME> [--host --port | --stdio] | authorize (--file|--key|--stdin) -- <cmd>` — SSH into a sandbox. `connect` opens a session; `serve` exposes the sandbox over SSH (host:port or stdio); `authorize` adds a pubkey to msb's SSH authz. Feature-gated (ssh).
- `msb inspect` `<NAME>` — Show detailed sandbox configuration and runtime status.
- `msb volume (alias: vol)` `subcommands: create | list(ls) | inspect | remove(rm)` — Manage named persistent volumes.
- `msb snapshot (alias: snap)` `subcommands: create | list(ls) | inspect | verify | remove(rm) | reindex | export | import` — Manage disk snapshots of stopped sandboxes; snapshots pin an image+upper-layer and can seed new sandboxes via `run --snapshot`.
- `msb install` `<IMAGE> [-n NAME] [resource/mount/env opts] [-f] [--list]` — Install a sandbox as a system command/alias (e.g. `msb install ubuntu` makes `ubuntu` launch a sandbox).
- `msb uninstall` `<NAME>` — Remove an installed sandbox command alias.
- `msb self update` `[flags]` — Update the `msb` binary and libkrunfw to the latest release (self-managed install, NOT an OS package).
- `msb sandbox` `(HIDDEN, internal)` — The internal subprocess entrypoint: the VMM/guest process that `msb run/create` re-execs. Not user-facing.
- `--tree (global)` `--tree` — Print the full command tree and exit (works even when required args are missing). Nice for discoverability/testing.

### Lifecycle model
Dual-mode, DB-backed (NOT engine-as-source-of-truth). State lives in a SQLite DB at `~/.microsandbox/db/msb.db` (sea-orm) tracking sandbox records, runs (PID), images, volumes, snapshots. EPHEMERAL: an UNNAMED `msb run` sandbox is removed after the foreground command exits (stop_and_wait + remove_persisted). PERSISTENT: NAMED sandboxes and `msb create` survive; reused on subsequent `run`/`exec`; explicitly removed via `remove`/`stop`. Lifecycle caps via `--max-duration` (hard kill) and `--idle-timeout` (stop after inactivity). REAPER: `spawn_reaper()` is fired fire-and-forget at the start of EVERY CLI command (one-shot, guarded by std::sync::Once). `reap_stale_sandboxes()` queries the DB for records claiming Running/Draining and reconciles each against the actual PID (`pid_is_alive`); dead processes are marked Crashed and cleaned. This is the backstop for VMM crashes (SIGSEGV/SIGKILL) that bypassed the normal stop path. `--replace`/`--replace-with-timeout` recreate an existing named sandbox (SIGTERM then SIGKILL after a default 10s).

### Isolation & flags
- Resources: -c/--cpus (vCPU count), -m/--memory (e.g. 512M, 1G). Maps to libkrun guest sizing — analogous to our krun.cpus/krun.ram_mib annotations.
- Mounts: -v/--volume SOURCE:DEST[:OPTIONS] (host path OR named volume), --tmpfs PATH[:SIZE[:OPTIONS]] (in-memory). Internal CliMountOptions support readonly, noexec, stat_virtualization, host_permissions, size_mib — i.e. ro/noexec ARE expressible per-mount.
- Pre-boot rootfs staging (host->guest, before boot): --copy/--copy-file/--copy-dir SRC:DST, --mkdir PATH, --rm PATH (hide/remove from guest rootfs). Also --script/--script-raw/--script-path register named scripts under /.msb/scripts (on PATH).
- Identity/exec context: -w/--workdir, --shell (default /bin/sh), -e/--env KEY=val, -H/--hostname (defaults to sandbox name), -u/--user (drop to nobody/uid), --entrypoint override, --init PATH|auto + --init-arg + --init-env (hand PID1 to systemd/init).
- Network (feature-gated `net`, OFF in our model): -p/--port HOST:GUEST[/udp] publishes; --no-net = deny-all egress (sugar for --net-default deny); --net-default/--net-default-egress/--net-default-ingress ACTION; --net-rule '<action>[:dir]@<target>[:proto[:ports]]' allowlist grammar with IP/CIDR/domain/suffix/group targets; egress DEFAULT=deny (implicit allow@public when no rules) but ingress DEFAULT=allow. Plus DNS rebind protection ON by default, --dns-nameserver, --max-connections, --trust-host-cas.
- Advanced net (the surface we DON'T want): --tls-intercept (MITM HTTPS proxy with custom CA), --tls-bypass, --no-block-quic, and --secret ENV@HOST / --on-secret-violation (block/block-and-log/block-and-terminate/passthrough) — explicit anti-exfiltration tooling.
- Image pull policy: --pull always|if-missing(default)|never; --oci-upper-size for writable overlay sizing.
- Per-exec POSIX rlimits: --rlimit nofile=1024,nproc=64,as=... (cheap blast-radius control independent of cgroups).
- Output: `status --format json` for machine-readable output; stable exit codes (run forwards the guest command's exit code; structured boot-error and exec-failed renderers map to specific codes).

### Standout features
- Docker-mirrored verb set (run/create/start/stop/ps/exec/cp/rm/logs/inspect/image/pull) — instantly familiar; uses visible_alias for ls/ps/rm/cp/vol/snap.
- --tree global flag dumps the entire command tree even with missing required args — great for docs/agent discovery.
- NO daemon and NO server subcommand in this version: `msb run` re-execs itself as the hidden `msb sandbox` VMM subprocess; the CLI process IS the supervisor. (The MCP server is a separate npm package, not a `msb server`.)
- Reaper-on-every-invocation pattern: cheap self-healing of crashed VMs without a long-running process.
- Snapshots as first-class lifecycle: snapshot a stopped sandbox, then `run --snapshot` boots a fresh VM seeded from it (golden-image workflow).
- Pre-boot rootfs mutation (--copy/--mkdir/--rm/--script) lets you stage a guest without building a custom image.
- `msb install <image>` turns a sandbox into a host command alias — neat UX for 'sandboxed python/ubuntu as a binary'.
- Agent integration is OUT-OF-TREE (skills/ and mcp/ are submodules pointing at separate repos: superradcompany/skills and microsandbox-mcp) — the core stays a clean CLI/SDK.
- SSH `serve --stdio` mode: one SSH transport over stdin/stdout, composes with external tunneling.

### Gotchas
- skills/ and mcp/ submodules are NOT checked out in this clone (empty dirs); they live in separate repos (superradcompany/skills, microsandbox-mcp). Couldn't read the actual Skill files here — the Agent-Skills teaching content is external.

- Ingress network default is ALLOW (only egress defaults to deny) — published ports are unfiltered unless you add ingress rules. A footgun if you assume --no-net covers inbound.
- Heavy: a Rust workspace of ~15 crates + an in-guest agentd + 3 language SDK bindings + MCP + skills. This is the graduation target, not a model to copy at bash scale.
- `msb run <named>` silently REUSES an existing sandbox and IGNORES creation flags (warns only) unless --replace — surprising if you expect a fresh VM each time. Our unnamed-ephemeral default avoids this ambiguity.

### Fedora fit
Low / non-canon. Runs on Linux-with-KVM (works on Fedora) and uses libkrun like us, but it is installed via `curl -fsSL https://install.microsandbox.dev | sh` and updated by `msb self update`, bundling its own libkrunfw — it is NOT a Fedora/dnf package and does NOT use podman+crun-krun. ### Lessons for our tool
- Verb naming: adopt the familiar Docker-ish core (run/exec/stop/rm/ps/logs/inspect). For US: `run` = ephemeral one-shot (--rm default), and make persistence an explicit opt-in (their NAMED-vs-unnamed split is exactly our 'ephemeral by default, persist is opt-in that stamps a label' decision — validates it). Consider a `--name`/`--keep` flag rather than a separate verb.
- Their reaper-on-every-invocation is the model for our label-driven reap BACKSTOP: run a cheap reconcile sweep at the top of every script invocation (query podman by label, hard-kill/clean any managed sandbox whose container is dead/orphaned). Crucially, because podman labels are our source of truth, we get this WITHOUT a DB — sidestepping the exact drift their SQLite reaper exists to fix. Call that out as a design win.
- `status --format json` + forwarding the guest's exit code is the right machine-readable contract. Implement our `--json`/stable-columns mode and propagate the workload's exit code verbatim; reserve distinct exit codes for our doctor/precondition failures vs workload failures.
- Per-mount ro/noexec and per-exec rlimits (nofile/nproc/as) are cheap, high-value blast-radius controls that fit an accident threat model perfectly — wire `--read-only`/noexec on our default mounts and a conservative rlimit set; they don't need cgroup complexity.
- Their network model is a cautionary tale of scope creep for OUR threat model: --net-rule grammar, TLS interception with a MITM CA, and --secret/--on-secret-violation are ANTI-EXFILTRATION features we explicitly do NOT want. Stay at `--network none` default + simple localhost port publish; do not build an allowlist DSL or a secret-leak detector. Note: their ingress-default-allow is a footgun — our 'none by default, publish to localhost explicitly' is safer.
- Pre-boot rootfs staging (--copy/--mkdir/--script) overlaps with our worktree-per-sandbox idea — but they let the user point at arbitrary host paths. Our locked decision (the TOOL creates the worktree, guards destructive git) is the safer variant; keep it.
- SSH: they bake in a full SSH server/authorize/serve stack. We instead forward the host ssh-agent so keys never enter the guest (smolvm idea) and stay orthogonal to remoting — simpler and a smaller attack surface. Their `serve --stdio` confirms keeping transport composable (we emit parseable localhost endpoints for ssh/Caddy to wrap).
- Keep agent integration OUT of the core, like they do (skills/mcp as separate repos): our thin Claude SKILL.md should wrap the bash script's stable CLI, never reach into engine internals — matches our no-plugin philosophy.
- This repo IS our graduation signal made concrete: ~15 Rust crates, a SQLite state layer with a reaper to reconcile it, a network policy DSL, a TLS-intercept proxy. When our single bash script starts needing concurrency-safe allocation or tested isolation guarantees, adopting/forking microsandbox (or a Rust rewrite in its shape) is the path — but we should resist importing its platform breadth (server, MCP, net DSL, multi-SDK) before then.
- Their `--max-duration` (hard kill) and `--idle-timeout` (stop-on-inactivity) are worth stealing as optional flags for long-lived agent webapps — a self-expiring sandbox is a good accident backstop on top of our trap+reap layers.

- `pack run` `(packed binary)` — Run a packed self-contained VM executable; main() auto-detects packed mode before normal CLI parsing.
- `pack push / pull / inspect / prune` `varies` — Push/pull packed artifacts to/from a registry, inspect metadata, prune cached packs.
- `config show` `` — Show smolvm configuration (registries, defaults).
- `config registries` `` — Manage registry configuration.
- `serve start` `[--json-logs] [tcp ADDR | unix PATH]` — Run an HTTP API server for programmatic control (daemon-style; installs SIGCHLD reaper for zombie VM children). NOT the default usage path.
- `serve openapi` `` — Emit the OpenAPI spec for the serve API.
- `_boot-vm` `CONFIG.json (hidden)` — Internal: boot a VM subprocess from a boot-config JSON. Not for direct use.
- `_cleanup-ephemeral` `VM_NAME PID START_TIME EPHEMERAL_NAME (hidden)` — Internal: detached helper that reaps an ephemeral VM after its command exits (kill + remove data dir + DB deregister). Spawned so the parent CLI can return the guest exit code immediately.

### Lifecycle model
Two explicit modes. (1) EPHEMERAL is the default for `machine run`: VM is destroyed after the command exits; `-d/--detach` or `machine create` opts into PERSISTENT named machines. (2) Cleanup is multi-layered and out-of-process: on normal `machine run` exit the parent CLI spawns a DETACHED `smolvm _cleanup-ephemeral <name> <pid> <start_time> <ephemeral_name>` helper, then exits with the guest's exit code immediately while teardown happens async. The helper enforces a strict ordering invariant: (a) SIGKILL only via PID+start_time verification (refuses to signal if start_time missing or PID reused), (b) remove data dir ONLY after process confirmed dead AND only if path passes is_safe_cache_path (non-empty, strictly inside cache root, not equal to root, not a symlink), (c) DB deregister ONLY after dir removal succeeds. Any failed step intentionally leaves state behind for a backstop ORPHAN SWEEP (cleanup_orphaned_ephemeral_vms) that runs per-command at startup (skipped for the ephemeral run path itself). State of record is a local DB (src/db.rs) + a hash-derived per-VM data dir, NOT podman labels. The `serve` daemon additionally installs a SIGCHLD handler to reap zombie VM children.

### Isolation & flags
- Network OFF by default: `--net` is required to enable any outbound access. This is the safe default.
- Egress allowlisting layered on --net: `--allow-cidr CIDR` (repeatable, implies --net), `--allow-host HOSTNAME` (repeatable, resolved at VM start, implies --net; resolution failure is a HARD ERROR to avoid silently weakening policy), `--outbound-localhost-only` (implies --net). Hostname allowlist is enforced via an in-guest DNS filter proxy bridged over its own vsock port (DNS_FILTER).
- Mounts: `-v/--volume HOST:CONTAINER[:ro|:rw]` repeatable; parsed as host:guest[:ro|:rw], default rw. Delivered as virtiofs devices tagged smolvm0, smolvm1,...; the read_only flag is propagated to the guest agent via env (SMOLVM_MOUNT_i=tag:guest_path:ro|rw). No mounts by default.
- Port publishing: `-p/--port HOST:GUEST` repeatable; duplicate-port check enforced.
- Resource caps default HIGH/dev-oriented: --cpus default 4 (DEFAULT_MICROVM_CPU_COUNT), --mem default 8192 MiB (8 GiB). Zero-valued resources rejected at create. Also --storage GiB and --overlay GiB disk sizing.
- GPU off by default: `--gpu` (Vulkan via virtio-gpu) + `--gpu-vram MiB` (default 4096).
- `--ssh-agent` (Security heading): forwards host SSH agent so keys never enter the guest (see standout).
- `--docker-config`: mounts ~/.docker config into the VM for registry auth (opt-in).
- Container lives in its own mount+env namespace inside the VM, so host-forwarded resources (ssh socket, env) must be explicitly re-injected into the OCI spec; nothing leaks implicitly.
- No persistent-label model; isolation/lifecycle state is tracked in a local DB + hash-derived data dir, not in the container runtime's metadata.

### Standout features
- SSH-agent forwarding with keys NEVER in the guest, implemented end-to-end over vsock (this is exactly the idea our project lifts). HOST side: if `--ssh-agent`/record.ssh_agent, resolve SSH_AUTH_SOCK from the host env (hard error with actionable message `Start an SSH agent with: eval $(ssh-agent) && ssh-add` if unset); register a vsock port `ports::SSH_AGENT` via `krun_add_vsock_port2(ctx, SSH_AGENT, host_socket_path, listen=false)` so the GUEST dials OUT and the HOST receives via that Unix socket (the host's real agent socket); also push env `SMOLVM_SSH_AGENT=1` into the VM. GUEST side (smolvm-agent/src/ssh_agent.rs): on boot if SMOLVM_SSH_AGENT=1, start() a background bridge thread that binds a Unix listener at /tmp/ssh-agent.sock (chmod 0777), sets SSH_AUTH_SOCK=/tmp/ssh-agent.sock in the agent process, and for each incoming connection opens a vsock connection to ports::SSH_AGENT and relays bytes bidirectionally using libc::poll() (30s idle timeout, since agent connections are short-lived). CONTAINER side: because the container has its own mount+env namespace, inject_into_container() bind-mounts /tmp/ssh-agent.sock into the container at the same path (rw) and sets SSH_AUTH_SOCK there (replacing, not duplicating, any stale value). Net effect: git/ssh inside the container talk to the host agent over vsock; private keys never cross into the VM.
- DNS-filter proxy bridged over its own vsock port for hostname egress allowlists.
- `.smolmachine` packed self-contained executables (pack create/run/push/pull) with appended assets / Mach-O section detection; main() reroutes to packed-binary mode before clap parsing.
- Smolfile (TOML) declarative machine config, mergeable with CLI flags.
- HTTP/OpenAPI control plane (`serve`) for programmatic use, with its own zombie-reaping.
- Detached, verification-gated ephemeral reaper as a hidden subcommand (clean separation of teardown from the user-facing exit path).

### Gotchas
- VENDORS FORKED libkrun and libkrunfw from github.com/smol-machines (git submodules in .gitmodules; prebuilt fork dylibs shipped in lib/, e.g. libkrun.dylib, libkrunfw.5.dylib). This is a custom kernel+VMM fork, NOT upstream containers/libkrun — do NOT adopt it; our project deliberately uses official Fedora podman+crun-krun+libkrun packages instead.
- Heavily macOS-oriented: lib/ ships .dylib + libMoltenVK/libepoxy/virglrenderer + a smolvm.entitlements file; Linux is a secondary target. Not Fedora-canon.
- Defaults are dev-machine-sized (4 vCPU / 8 GiB RAM), much larger than a conservative blast-radius sandbox would want.
- It is its own VMM/agent stack (embedded kernel, guest agent, custom protocol crate) — a parallel runtime, not a podman wrapper; large surface, not single-file-auditable.
- Maintains its own state DB + hash-derived data dirs and an orphan-sweep, i.e. exactly the state-file/daemon machinery our project explicitly avoids by using podman labels as source of truth.
- Guest bridge socket is chmod 0777 (all VM users can reach the forwarded agent) — acceptable single-user, but worth noting it widens access inside the guest.
- `machine exec` runs at VM (Alpine rootfs) level, NOT inside the container — easy to confuse with `run`'s container context.

### Fedora fit
Low / non-canon. It ships and depends on FORKED libkrun + libkrunfw (smol-machines forks) as vendored prebuilt libraries, is primarily a macOS/Hypervisor.framework tool (dylibs, MoltenVK, entitlements), and is its own VMM+agent stack rather than anything built on Fedora's podman/crun-krun packaging. Useful as a design/UX reference only; not as a packaging or runtime model.

### Lessons for our tool
- Verb-naming prior art to learn from: smolvm uses `machine run` (ephemeral default), `machine exec`, `machine create/start/stop`, `machine delete` (alias rm), `machine ls` (alias list), `machine shell` (alias sh), `machine cp`, `machine status`. Ephemeral-by-default on `run` matches our locked decision; consider whether we want the `machine` noun prefix or flat verbs.
- Adopt their SSH-agent forwarding architecture conceptually: resolve host SSH_AUTH_SOCK, hard-fail with an actionable message if unset, forward the host agent socket into the guest, set SSH_AUTH_SOCK inside, and rely on the relay so PRIVATE KEYS NEVER ENTER THE GUEST. With our podman+krun engine the transport differs (use a bind-mount of $SSH_AUTH_SOCK into the container with `-v $SSH_AUTH_SOCK:...:ro` style + `-e SSH_AUTH_SOCK=...` rather than a custom vsock bridge), but the key-never-in-guest invariant and the 'inject both the socket file AND the env var because the container namespace is separate' lesson carry over directly.
- Their three-step verification-gated reaper validates our three-layer cleanup plan: (1) kill ONLY after PID+start_time verification (avoid killing a reused PID), (2) delete data ONLY after confirming the process is dead AND the path passes a strict safety check (inside our managed root, not a symlink, not empty/root), (3) deregister last. We can mirror this ordering even though podman labels (not a DB) are our source of truth, and the path-safety guard is a must for any rm -rf in the reaper.
- Make the reaper out-of-process / non-blocking like their detached `_cleanup-ephemeral`, so the user gets the workload's exit code immediately while teardown happens in the background. A hidden internal subcommand is a clean pattern for our label-driven reap backstop.
- Mirror their safe defaults: network OFF unless explicitly enabled (`--net`), no mounts unless asked, GPU off — these match our --network none default and read-only-where-possible stance. But set CONSERVATIVE resource defaults (NOT their 4 vCPU / 8 GiB); use krun.cpus/krun.ram_mib annotations with smaller blast-radius caps.
- Their egress allowlist (--allow-host with hard-fail on resolution, --allow-cidr, --outbound-localhost-only) is more than our threat model (accident, not adversary) needs — we are orthogonal to network/Tailscale and publish to localhost, so we should NOT build a DNS-filter proxy; keep `--network none` default and let composition happen outside.
- Adopt their `cp` ergonomics: `machine:path` syntax on either side and send progress to STDERR so redirects stay clean — good fit for our machine-readable-output requirement.
- Validate resources at create-time, not start-time (they explicitly fixed a bug where zero resources only failed at start). Our centralized sandbox-spawn function should validate/normalize caps before launching.
- Confirmation that growing this into a parallel VMM+agent+DB+forked-libkrun stack is exactly the complexity our single-bash-script-over-podman approach avoids. Their DB/orphan-sweep/state machinery is the kind of thing that signals 'graduate to Rust' for us; staying label-driven on podman keeps us single-file-auditable.
- Do NOT adopt smolvm's forked libkrun/libkrunfw or its packed `.smolmachine`/macOS dylib distribution model — it is non-Fedora-canon and off-mission.

## agent-sandbox (thomaspeklak/agent-sandbox) — binary name `ags`

**Tier:** inspiration  |  **Language:** Rust (crate `ags`, hand-rolled arg parser in crates/ags/src/cli.rs — NOT clap). Guard policies in YAML; container entrypoint in bash; TS guard extension for Pi.

**Purpose:** Launches interactive AI coding agents (pi/claude/codex/gemini/opencode/shell) inside a rootless Podman CONTAINER with hardened defaults, host-service bridges (auth proxy, clipboard, host UI), and a destructive-command guard. Sandboxes the AGENT HARNESS itself, not just a workload — the opposite scoping from ours.


### Verb surface

- `(default run mode)` `--agent <pi|claude|codex|gemini|opencode|shell> [--browser] [--tmux] [--stop-when-done] [--psp] [--psp-keep] [--yolo] [--root] [--lockdown] [--wayland-compositor-passthrough] [--defaults|-D] [--config PATH] [--add-dir|-d PATH ...] -- [agent args...]` — No verb — run is the implicit subcommand selected by presence of --agent. Builds a launch plan and execs `podman run --rm -it`. Args after `--` pass through to the agent CLI. This is the entire 'create a sandbox' surface; there is no `run`/`start`/`create` noun-verb.
- `setup` `(none)` — One-time bootstrap: generates SSH auth + signing keypairs, prints pubkeys, ensures Pi guard/settings assets exist, optionally stores secrets via secret-tool.
- `doctor` `(none)` — Verb-zero precondition/health check: required+optional host tooling, config/asset presence, tool binaries+mounts, IMAGE PRESENCE, whether bundled `dcg` exists inside the image (warns 'guards will fail open'), SSH keys + dedicated ssh-agent state, secret sources, session dir writeability, browser setup.
- `update-image` `[--keep-existing]` — Rebuilds the sandbox image from the configured Containerfile, pulling latest release tags of bundled tools (br/bv/dcg) with checksum verification; removes the previously-tagged image after success unless --keep-existing. Does NOT touch agent CLIs.
- `update` `[--keep-existing]` — DEPRECATED ALIAS for update-image (kept as SubCommand::UpdateDeprecated in parser).
- `update-agents` `(none)` — Installs/updates agent CLIs (pi, codex, gemini, opencode, claude) into PERSISTENT volumes via a throwaway container. pnpm with ignore-scripts=true, stable store, strips stale self-update + npm-global shims. Driven by [update] config.
- `install` `[--link-self] [--force] [--add-agent-mounts]` — Writes baseline assets (~/.config/ags/Containerfile, tmux.conf, agent guard.ts + settings.json); --link-self symlinks ~/.local/bin/ags; --add-agent-mounts appends default [[agent_mount]] entries to config.toml.
- `uninstall` `(none)` — Reserved no-op today.
- `create-aliases` `[--mode wrappers|aliases|both] [--shell fish|zsh|bash] [--force]` — Generates managed wrapper scripts in ~/.local/bin and/or delimited alias blocks in shell rc files; uses --defaults so direct launch and wrappers stay in sync.
- `completions` `--shell bash|zsh|fish` — Prints shell completion script to stdout.
- `config` `(none)` — Present in parser (SubCommand::Config) — launches a TOML config editor / shows merged effective config. NOT documented in COMMANDS.md (undocumented surface).

### Lifecycle model
EPHEMERAL-by-default and hard-wired: every run is `podman run --rm -it` (crates/ags/src/podman/args.rs:13 — `--rm` is unconditional, no persist flag). Container dies on exit; nothing reaps because nothing survives. PERSISTENCE is NOT a container property here — it lives in named volumes (agent homes, pnpm store) populated by `update-agents`, so agent installs/state persist across ephemeral runs. Sidecars (auth proxy, PSP, clipboard, host UI, webview relay) are per-run host processes torn down when the container exits; PSP gets SIGTERM then SIGKILL after 5s to clean its child containers (--psp-keep retains them, swept on next PSP start). Per-run temp dirs (auth-proxy socket/shim) cleaned on container exit. No daemon, no state file, no age-based reaper — the `--rm` + process-tree teardown IS the whole model. Container names are per-run (plan.container_name).

### Isolation & flags
- DEFAULT SecurityConfig (plan/types.rs:110): --userns=keep-id (rootless, host UID maps in), --security-opt=no-new-privileges, --security-opt=label=disable (SELinux OFF — relevant on Fedora!), --cap-drop=all, --pids-limit=4096. No tmpfs by default.
- Base flags always: run --rm -it --pull=never (offline; image must pre-exist, doctor/update-image build it).
- NETWORK default = slirp4netns:allow_host_loopback=false (NOT --network none). With --browser it flips to allow_host_loopback=true. So network is ON by default but host loopback is blocked unless browser mode. Host services reached via host.containers.internal, never 'localhost' (localhost is container-local).
- --lockdown SecurityConfig adds hardened tmpfs: /tmp rw,nosuid,nodev,size=1g; /var/tmp +noexec size=256m; /run +noexec size=64m. Disables secrets, passthrough env, ssh-agent, git config, generic [[mount]]/[[tool]] mounts, sidecars; stages a sanitized ephemeral agent home and discards session history on exit. Network STAYS ON in lockdown. Mutually exclusive with --browser/--psp/--root/--wayland.
- --root SecurityConfig (plan/types.rs:141): DROPS keep-id (UID0->host user), --user=root, removes no-new-privileges, cap-drop=NONE — escape hatch for installing packages; keeps only label=disable.
- SSH: keys NEVER enter the container — a DEDICATED ssh-agent is spawned per session (ssh.rs) and the SSH_AUTH_SOCK is forwarded. (This is the idea our project already lifted from smolvm.)
- Mounts: explicit [[mount]]/[[agent_mount]]/[[tool]] config entries; README pushes 'prefer ro mounts', 'only mount what the agent needs'. --add-dir/-d adds same-path mounts for one run.
- Auth proxy: $BROWSER inside container points at a shim; URL-opens are relayed over a Unix socket to a host dialog requiring explicit approval (never silent); OAuth loopback callbacks relayed back so the container never binds host ports.

### Standout features
- doctor / update-image / update-agents as cleanly separated verbs — image lifecycle vs agent-binary lifecycle are deliberately split (update-image rebuilds image; update-agents fills persistent volumes). Strong prior art for OUR doctor verb-zero.
- doctor explicitly probes whether `dcg` is inside the IMAGE and warns that guards 'fail open' if missing — a precondition check tied to a security guarantee.
- Dedicated per-session ssh-agent forwarding so private keys stay on host.
- Auth-proxy/host-dialog pattern: sandboxed agent cannot open URLs or bind host ports without an explicit host-side approval dialog.
- Project-local dcg policy packs (.dcg/packs/*.yaml) — declarative regex guards for destructive shell commands, shipped IN the repo and consumed inside the sandbox.
- --lockdown as a single, well-defined hardened profile (sanitized ephemeral home + tmpfs + sidecars off) for untrusted/foreign repos.
- Worktree-parent detection in git.rs (worktree_parent_repo_dir / repo_root) so the tool mounts the shared .git common dir for linked worktrees — solves the orphaned-from-.git problem we care about.
- Machine-friendly distribution: PKGBUILD + completions + create-aliases wrappers.

### Gotchas
- git worktree prune GOTCHA (the named one): the sandbox cannot see all host worktree paths, so `git worktree prune` from inside treats unseen worktrees as stale and deletes their metadata from .git/worktrees, ORPHANING valid host worktrees. They block it via dcg policy .dcg/packs/git-worktree-sandbox.yaml (regex severity:critical). Direct lesson: our worktree-per-sandbox design must guard `git worktree prune` (and similar) inside the guest.
- --security-opt=label=disable means SELinux confinement is turned OFF for the container — on Fedora this discards a real host-protection layer. We should evaluate keeping SELinux labeling (:Z/:z mounts) rather than disabling.
- Network is NOT off by default (slirp4netns with loopback blocked) — different from our locked --network none default. allow_host_loopback toggles only with --browser.
- --rm is unconditional in the parser/args — there is genuinely NO persist-the-container path; 'persistence' is volumes only. If you expect a stopped container to inspect, you must use --tmux to drop to a shell.
- guards 'fail open' if dcg is absent/errors OUTSIDE lockdown; only IN lockdown does Bash classification fail CLOSED. So the safety guarantee is conditional on the run mode.
- CLAUDE.md in the clone is a symlink to AGENTS.md which is absent — the canonical agent doc didn't clone; COMMANDS.md + cli.rs are the authoritative sources.
- `config` subcommand exists in the parser but is undocumented in COMMANDS.md.
- --root removes cap-drop and no-new-privileges entirely — a wide escape hatch that silently undoes the hardening; easy to leave on.

### Fedora fit
NOT Fedora-canon. It is an ARCH project: ships PKGBUILD.in + .install (Maintainer email, AUR-style), depends on git/openssh/podman, built with cargo. No Fedora RPM/spec, no copr. Engine is plain rootless Podman with the DEFAULT OCI runtime (crun) — there is NO krun/libkrun/microVM anywhere; this is podman CONTAINER tier, weaker isolation than our microVM. The label=disable (SELinux off) default is actively un-Fedora-friendly. Useful as CLI/UX prior art, not as a packaging or isolation model.

### Lessons for our tool
- VERB NAMING: their surface is `doctor`, `update-image`, `update-agents`, `setup`, `install`, `uninstall`, `create-aliases`, `completions` — and crucially NO `run`/`start` verb (run is implicit via --agent). For OUR scriptable flag-preceded design we likely DO want an explicit run/up verb, but their split of doctor / update-image / update-agents validates separating 'check preconditions' from 'rebuild image' from 'refresh contents'. Consider stealing `doctor` verbatim as our verb-zero name.
- doctor as a real gate: copy their pattern of doctor checking IMAGE PRESENCE and a specific in-image security dependency. Our doctor should check KVM access, crun built with +LIBKRUN, crun-krun present — same spirit of 'a precondition tied to a guarantee'.
- WORKTREE: their git.rs proves you must resolve the worktree's shared .git common dir and mount it, or the agent is orphaned from git — exactly our 'tool creates the worktree' decision. AND adopt their concrete guard: block `git worktree prune` (and destructive git) inside the guest; a regex/wrapper is enough.
- SSH-agent forwarding is independently arrived-at here too (dedicated per-session agent, forward SSH_AUTH_SOCK) — confirms our smolvm-derived decision is sound and standard.
- DON'T disable SELinux. Their label=disable default is the anti-pattern for us on Fedora; prefer :Z/:z labeled, read-only-where-possible mounts so the host MAC layer stays intact.
- NETWORK default: they chose loopback-blocked-but-on; we chose --network none. Ours is stricter and correct for blast-radius containment of an untrusted WORKLOAD. Keep --network none default; make 'enable network' an explicit flag (their --browser is the analog).
- EPHEMERAL: their unconditional --rm validates our --rm-by-default + persistence-as-explicit-opt-in. But note they have NO label-driven reaper because nothing survives — OUR reap backstop only matters because we allow opt-in persistence; keep the reaper scoped to labeled persistent sandboxes.
- ESCAPE-HATCH HYGIENE: their --root nukes all hardening at once. If we add an analogous escape hatch, make it loud, single-run, and stamped in a label so the reaper/doctor can see it.
- SCOPE DIFFERENCE to keep clear: agent-sandbox sandboxes the AGENT HARNESS (mounts agent homes, forwards auth, host UI). We sandbox the WORKLOAD the agent produces. Resist importing their auth-proxy/clipboard/host-UI sidecar complexity — that surface exists because they run the harness inside; it is out of scope for us and would bloat the single bash script.
- GRADUATION: this Rust crate is roughly what we'd become if we outgrew bash — note the hand-rolled parser (not clap) is already ~400 lines for ~10 verbs; that volume of arg-parsing pain is itself the graduation signal our context names.
- Offline by default (--pull=never) + image built/verified by update-image with checksums is a good supply-chain default to mirror.

## ERA (binsquare/era) — era-agent CLI. DEPRECATED, succeeded by smol-machines/smolvm.

**Tier:** inspiration  |  **Language:** Go (CLI binary named `agent`); bash setup scripts; krunvm + buildah shelled out as subprocesses.

**Purpose:** Run untrusted / AI-generated code locally with microVM isolation (krunvm/libkrun) but container-like ergonomics. A Go `agent` CLI wraps `krunvm` create/start/delete, mapping language names to public Docker images, with persistent VM metadata in BoltDB plus an optional Cloudflare Worker hosted layer.


### Verb surface

- `agent vm create` `--language <python|javascript|node|typescript|ruby|go|golang> [--image <override>] --cpu <n=1> --mem <MiB=256> --network <none|allow_all, default none> [--persist]` — Provisions a long-running named VM: resolves a rootfs image from the language, runs `krunvm create --name <id> --cpus --mem [--volume ...] <image>`, sets status ready, persists a VMRecord to BoltDB. Does NOT start/execute. ID is `<lang>-<unixnano>`.
- `agent vm run` `--vm <id> --cmd "..." [--file ./x] --timeout <seconds, REQUIRED >0>` — Runs one command in an existing VM via `krunvm start <id> -- /bin/bash -c 'echo <base64> | base64 -d | bash'`. Captures stdout/stderr to files under the VM's /out. Auto-relaunches the VM if krunvm reports it missing or it was stopped. Logs result; does NOT print captured output to console.
- `agent vm exec` `--cmd "..." [--file ./x] [--timeout <seconds=30>] (--vm <id> repeatable | --all)` — Like run but fan-out: targets one-or-more VMs or all ready/running VMs. Same base64 bash exec path. UNLIKE run, it streams the captured stdout/stderr files to the console. Aggregates per-VM errors with errors.Join.
- `agent vm shell` `--vm <id> [--cmd /bin/bash (default)]` — Interactive shell into a VM: `krunvm start <id> -- <cmd...>` wired directly to os.Stdin/Stdout/Stderr. Splits --cmd on whitespace (naive Fields, no real quoting).
- `agent vm temp` `--language <=python> [--image] --cmd "..." [--file] [--timeout=30] --cpu <=1> --mem <MiB=256> --network <none|allow_all> [--persist]` — Ephemeral one-shot: creates a VM, runs the command, prints output, then ALWAYS cleans up the VM (even on failure path). This is the disposable-execution verb.
- `agent vm list` `[--status <state>] [--all]` — Tabwriter table of VMRecords (ID, Language, Status, CPU, Mem, Persist, Created, LastRun). Reconciles cached status against live `krunvm list`: marks absent VMs stopped, present-but-stopped VMs ready. Hides stopped unless --all or a --status filter is given.
- `agent vm stop` `(--vm <id> repeatable | --all)` — `krunvm delete <id>` then marks the record stopped (treats 'not found' as already-stopped). Note: stop and clean both ultimately call krunvm delete; stop keeps the BoltDB record + host storage, clean removes them.
- `agent vm clean` `(--vm <id> repeatable | --all) [--keep-persist]` — Tears down: `krunvm delete`, removes host storage Root dir, removes the persist dir unless --keep-persist, deletes the BoltDB record and cache entry.
- `agent server` `[--addr :8080]` — Hidden mode (not in usage text): starts an HTTP API server reusing the same VMService primitives (used by the Cloudflare layer).
- `global flags` `--log-level <debug|info|warn|error> | --log-file <path> | --vm-runtime <krunvm|libkrun>` — Parsed before the subcommand. Backend selectable (defaults krunvm). Env equivalents: AGENT_LOG_LEVEL, AGENT_LOG_FILE, AGENT_VM_RUNTIME.

### Lifecycle model
Two-tier. PERSISTENT by default for create/run/exec/shell: VMs are named, tracked in a BoltDB state DB (`/var/lib/agent/agent.db`, override AGENT_STATE_DIR), and survive across CLI invocations — the CLI is stateful (a state file/db IS the source of truth, opposite of our podman-labels design). EPHEMERAL is a dedicated verb (`vm temp`) that create→run→clean in one shot and always cleans up via defer-like inline cleanup even on the error path. Teardown is explicit (`stop` = krunvm delete + keep record; `clean` = delete + purge storage + drop record). NO reaper / NO age-based GC / NO trap-on-signal cleanup — a crashed `agent` leaves orphaned krunvm instances and BoltDB rows, reconciled lazily on the next `vm list` (present/absent → ready/stopped). `--persist` adds a host-side /persist volume dir that clean keeps only with --keep-persist. Concurrency guard is a crude single-instance lock: at startup it greps `ps -eo pid,command` for another process whose argv0 looks like `agent` and refuses to run.

### Isolation & flags
- Resource caps map straight to krunvm flags: --cpu→`--cpus`, --mem (MiB)→`--mem`. Defaults cpu=1, mem=256MiB.
- --network <none|allow_all> default none — BUT this is essentially COSMETIC: NetworkMode is stored in the record and never passed to krunvm (the launcher emits no network arg; krunvm of that era had no network-isolation knob). So 'network none' did NOT actually disable networking.
- Guest volumes are DISABLED by default (security/portability): /in, /out, /persist are only bind-mounted when AGENT_ENABLE_GUEST_VOLUMES=1. Consequence: `--file` staging requires that env var or it errors.
- Mounts when enabled: host <state>/vms/<id>/in→/in, .../out→/out, persist dir→/persist, passed as `krunvm --volume host:guest`. Shared dirs chmod 0777.
- ReadOnlyRoot=true is set on the StorageLayout record but (like network) is not wired into a krunvm flag in this launcher — aspirational metadata, not enforced.
- Guest runs as fixed UID/GID 65532 (defaultGuestUIDGID) per the layout, though again not explicitly enforced via a krunvm user flag here.
- Command injection avoidance: commands are base64-encoded host-side and `base64 -d | bash`-decoded in-guest to dodge quoting/escaping bugs.
- Container storage is forced to driver=vfs with a generated storage.conf, an insecureAcceptAnything policy.json, and a docker.io registries.conf, all under the state dir so buildah never touches root-owned /etc/containers.
- macOS-specific: requires a dedicated case-sensitive APFS volume; setup script verifies case-sensitivity and sets DYLD_LIBRARY_PATH to the Homebrew lib dir so krunvm finds libkrun.
- Per-exec timeout via context.WithTimeout (required >0 for run; default 30s for exec/temp).

### Standout features
- Language→image abstraction: `--language python` resolves to docker.io/library/python:3.11-slim, node:20-slim, ruby:3.2-slim, golang:1.22-bookworm; `--image` overrides. Multiple rootfs candidates supported with fallback-on-launch-failure.
- `vm temp` is a clean, copyable pattern for our ephemeral verb: create+run+print+guaranteed cleanup in one command, cleanup runs even when the workload exits nonzero.
- exec fan-out (`--all` / repeatable `--vm`) with per-target error aggregation (errors.Join) and a stable selection-dedup pattern.
- base64 command transport into the guest to sidestep shell-quoting bugs — a genuinely useful trick for passing arbitrary user code through `... -- /bin/bash -c`.
- Pluggable backend (`--vm-runtime krunvm|libkrun`) behind a VMLauncher interface — clean seam, relevant since we standardize on one runtime (crun --runtime=krun).
- Lazy state reconciliation in `vm list` (live `krunvm list` vs cached records) instead of a daemon.
- Self-bootstrapping container storage/policy/registries config written under the state dir, avoiding root-owned /etc/containers.

### Gotchas
- The headline security flag (`--network none`) was a NO-OP in the krunvm launcher — stored but never enforced. A would-be operator could believe a VM is network-isolated when it is not. Lesson: never let an isolation flag exist without wiring it to an actually-enforcing engine arg, and verify it.
- ReadOnlyRoot and the 65532 UID are likewise recorded-but-not-enforced metadata.
- No reaper/no signal trap: a killed `agent` orphans krunvm VMs and BoltDB rows until the next list reconciles status (and even then only flips status, doesn't delete).
- Single-instance lock is a `ps | grep agent` substring match — fragile, false-positives on any process path containing 'agent' (their comment even calls out logioptionsplus_agent), and serializes the whole tool to one concurrent invocation.
- `vm shell --cmd` splits on whitespace via strings.Fields → no real quoting; multiword/quoted shell commands break.
- run and exec diverge confusingly: run does NOT print captured output, exec does; both write to /out logs.
- Heavy macOS coupling (case-sensitive APFS volume, DYLD_LIBRARY_PATH, diskutil); Linux setup is hand-wavy ('may require kernel modules or specific privileges').
- Stateful BoltDB + cache can drift from reality (the whole missing-VM-recreate and status-reconcile machinery exists to paper over this).
- Pulls public docker.io images with insecureAcceptAnything policy — fine for a demo, a supply-chain footgun for anything real.
- Project is DEPRECATED; treat as directional prior art, not a maintained baseline.

### Fedora fit
Low-to-medium. It is a third-party Go binary (Homebrew tap `binsquare/era-agent-cli` + `slp/krun`) that shells out to krunvm + buildah — none of which are Fedora-canon. krunvm is a separate orchestrator (not `podman run --runtime=krun`), so the engine choice is explicitly NOT our stack. The README's Linux instructions are an afterthought; everything substantive (case-sensitive APFS, diskutil, DYLD_LIBRARY_PATH) is macOS-only. The underlying libkrun/libkrunfw IS the same tech Fedora packages, but ERA reaches it via krunvm rather than crun-krun. So: relevant for verb/ergonomics inspiration, not for packaging or runtime canonicity.

### Lessons for our tool
- Verb naming: their `temp` is exactly our ephemeral-by-default workload runner — but we should make ephemeral the DEFAULT of the primary `run` verb (with --rm), not a separate `temp` subcommand, and make persistence the explicit opt-in (a `--persist`/`--keep` flag that stamps a label). Borrow the create / run / exec / shell / list / stop / clean vocabulary; consider collapsing run+exec (the run-doesnt-print vs exec-prints split is a UX trap).
- Their `--network none` being unenforced is the single biggest cautionary tale: our centralized birth function must pass `--network none` to a runtime that ACTUALLY enforces it (podman does) and our `doctor` verb should assert it. Never ship an isolation flag that is only metadata.
- Reject their stateful model: they need a BoltDB + cache + ps-grep lock + lazy reconcile precisely because state lives outside the engine. Our podman-labels-as-truth design avoids all of that machinery — keep it. No state file, no daemon, no single-instance lock.
- Add what they LACK: a signal trap (INT/TERM/EXIT) plus a label-driven reap backstop. ERA orphans VMs on crash; our three cleanup layers (--rm, trap, reap) directly fix this gap.
- Steal the base64 command-transport trick for passing arbitrary agent-written code through `podman run ... -- bash -c` without quoting bugs.
- Steal language→image resolution with `--image` override and candidate fallback as an ergonomic convenience, but pin to Fedora/registry images we trust (NOT insecureAcceptAnything docker.io).
- Match their resource-cap ergonomics (--cpu/--mem with sane small defaults like 1 cpu / 256MiB) but route through krun.cpus / krun.ram_mib annotations on `podman run --runtime=krun`.
- Make captured output / list machine-readable: ERA only has a tabwriter table and file-logged stdout — we committed to --json/stable-columns + stable exit codes, which is the right call; their per-target errors.Join aggregation pattern is worth copying for multi-sandbox ops.
- Their `vm list` reconcile logic argues FOR our label-query approach: `podman ps --filter label=sandbox.managed-by` gives reconciliation for free with no DB.
- Guest-volumes-disabled-by-default is a good instinct; for us, default to read-only mounts and create the worktree ourselves (ERA bind-mounts host /in,/out, we should mount a tool-created git worktree and guard destructive git inside).
- SSH-agent forwarding (our smolvm-derived idea) is absent here — keys never enter the guest is still our differentiator; ERA has no secrets story beyond env-var recipes.
- Pluggable-backend interface (VMLauncher) is over-engineering for us since the engine is fixed at podman --runtime=krun; resist that abstraction until the graduation signal hits.

## arrakis (abshkbh/arrakis)

**Tier:** inspiration  |  **Language:** Go (host daemon, CLI, guest daemons); Python SDK (py-arrakis, external); separate MCP server repo

**Purpose:** A self-hosted platform to spawn/manage MicroVM sandboxes (cloud-hypervisor + Ubuntu rootfs) for AI-agent code execution and computer use, with snapshot-and-restore for agent backtracking. Exposes a REST API, Python SDK, and MCP server so humans and agents can programmatically run untrusted code, upload files, and snapshot state.


### Verb surface

- `start` `-n/--name (required), -k/--kernel, -r/--rootfs, -e/--entry-point, -s/--snapshot` — Create+boot a VM (POST /v1/vms). If --snapshot given it actually RESTORES (kernel/rootfs ignored). Returns JSON metadata (ip, status, tapDeviceName, portForwards).
- `stop` `-n/--name (required)` — Stop a running VM (PATCH /v1/vms/{name} status=stopped).
- `destroy` `-n/--name (required)` — Destroy/delete one VM (DELETE /v1/vms/{name}).
- `destroy-all` `(none)` — Destroy every managed VM (DELETE /v1/vms).
- `list` `-n/--name (required)` — Show info for one VM (GET /v1/vms/{name}): status, ip, tap device, port forwards.
- `list-all` `(none)` — List all VMs (GET /v1/vms).
- `snapshot` `-n/--name (required), -i/--id` — Create a snapshot of a VM (POST /v1/vms/{name}/snapshots), returns snapshotId.
- `restore` `-n/--name (required), -i/--id (required)` — Restore a VM from a snapshot (delegates to start with snapshotId). Note: restored VM reuses original IP, so original must be stopped/destroyed first.
- `pause` `-n/--name (required)` — Pause a running VM (PATCH status=paused).
- `resume` `-n/--name (required)` — Resume a paused VM (PATCH status=stopped — reused enum value, quirk).
- `upload` `-n/--name (required), -f/--file 'source,destination' (repeatable, required)` — Upload host files into the guest (POST /v1/vms/{name}/files); reads file content host-side and ships it as a JSON string.
- `download` `-n/--name (required), -p/--path (repeatable, required)` — Download files from guest (GET /v1/vms/{name}/files?paths=comma,list); prints content to stdout.
- `run` `-n/--name (required), -c/--cmd (required)` — Execute a shell command inside the guest (POST /v1/vms/{name}/cmd, blocking=true); prints output or returns error+output.

### Lifecycle model
PERSISTENT-by-default and daemon-owned. Every VM's lifetime is tied to the lifetime of the arrakis-restserver daemon ("Every VM started is managed by this server"). VMs are named, long-lived, and survive across client invocations; there is NO --rm/ephemeral default and no auto-teardown on client exit (the Python SDK's context-manager destroy() is the only ephemeral-ish convenience, implemented client-side). Cleanup is explicit via stop/destroy/destroy-all, OR forced via a process reaper: reapProcess() sends a graceful signal then hard-kills after reapVmTimeout=20s; used both for normal VM stop and during VM-process teardown. On restserver startup it aggressively wipes prior host state (cleanupTapDevices removes all tap* interfaces, cleanupBridge deletes br0, cleanupAllIPTablesRulesForIP flushes rules) — a global reset, not a per-sandbox label-driven reap. State lives in a host state_dir (./vm-state) plus in-memory server maps; there is no podman-style label-as-truth model.

### Isolation & flags
- VMM is cloud-hypervisor over /dev/kvm (Linux-only); each sandbox is a full MicroVM running Ubuntu, NOT a container — strong isolation but heavy.
- Root filesystem protected via overlayfs per sandbox (Dockerfile-built ext4 rootfs image).
- Memory cap is PERCENTAGE-of-host based: guest_mem_percentage in config (config.yaml=30, code default defaultGuestMemPercentage=50); calculateGuestMemorySizeInMB reads /proc host memory and allocates a fraction. No per-sandbox absolute --memory/--cpus flag exposed on the CLI.
- stateful_size_in_mb: 2048 — a fixed writable disk size baked into config, not a CLI knob.
- Networking is a SHARED L2 setup, NOT isolated-by-default: a host Linux bridge br0 (10.20.1.1/24), per-VM tap device added to the bridge, guest IPs from 10.20.1.0/24, plus iptables MASQUERADE/FORWARD ACCEPT rules and net.ipv4 forwarding=1 — i.e. guests get outbound internet by default. The opposite of --network none.
- Port forwarding is automatic and opinionated: config port_forwards expose guest 5901 (VNC 'gui') and 5736-5740 ('code') from the public server to each sandbox; host ports allocated dynamically (portallocator).
- SSH into guest with hardcoded credentials user=elara / password=elara0000 (baked in rootfs Dockerfile) — keys/secrets live in the image, NOT forwarded from host.
- Allocators are stateful host-side singletons (ipallocator, cidallocator/vsock, portallocator, 'fountain' tap allocator) — concurrency-managed allocation that a daemon needs but a single bash script does not.
- Annotations/labels: none in the podman sense; identity is the VM 'name' string tracked by the daemon's in-memory maps + state_dir, which is exactly the state-file/daemon coupling our project rejects.

### Standout features
- Snapshot-and-restore of full VM state (processes, files, memory) for agent backtracking / MCTS — the headline feature; cloud-hypervisor pause+snapshot under the hood.
- VNC server + Chrome pre-installed in every guest for 'computer use', with auto host port-forwarding so the GUI is reachable without extra setup.
- Three programmatic surfaces layered on one engine: REST API (OpenAPI server-api.yaml), Python SDK py-arrakis (PyPI), and a separate MCP server repo for Claude Desktop/Cursor/Windsurf.
- vsock-based control plane: host talks to in-guest cmdserver over vsock (cmd/vsockclient, cmd/vsockserver, cidallocator) for run/upload/download.
- Dockerfile-based rootfs customization and prebuilt kernel, with option to pass your own kernel at start.

### Gotchas
- restserver must run as root (sudo) and performs destructive global host setup on boot: it deletes ALL tap* interfaces and the br0 bridge and flushes iptables — co-tenant-hostile, will stomp unrelated networking.
- restore reuses the original VM's IP, so you must stop/destroy the original first or you get a collision — a sharp edge the README itself flags.
- resume reuses status=stopped in the PATCH enum to mean resume (the enum is [stopped, paused]); semantically muddy verb/state overloading.
- Single-host, single-daemon SPOF: kill the daemon and you lose management of every VM; VM lifetime coupled to daemon lifetime.
- Guest auth is a hardcoded password baked into the image — fine for a toy, a real secret-management gap.
- Default networking gives guests outbound internet + NAT, so untrusted agent code can phone home by default; isolation is VM-boundary only, not network-egress.
- AGPLv3 + commercial-license/CLA — copyleft, not freely vendorable into another project.

### Fedora fit
Poor. Not packaged in Fedora. Engine is cloud-hypervisor (not podman/krun) installed via curl|bash of prebuilt binaries and resources, targeted at Ubuntu guests and GCP/GCE host setup. Requires running a root daemon and mutating host bridge/iptables globally. Nothing here maps to 'official Fedora packages, no daemon, no third-party runtime.' Useful only as conceptual prior art, not as a dependency.

### Lessons for our tool
- This repo IS the platform-creep anti-pattern to avoid: a root host daemon (arrakis-restserver) + in-guest daemons (cmdserver, guestinit/vsockserver) + Go CLI + Python SDK + external MCP server, all to do what `podman run --runtime=krun` already does. Our value is staying ONE bash script + thin SKILL.md with podman as the only source of truth — resist every pull toward a daemon, REST layer, SDK, or MCP.
- Verb naming to LEARN FROM: keep the obvious, well-understood ones — start, stop, list, destroy. Their 'list' (one) vs 'list-all' split is clunky; prefer a single `list` with optional name + a --json flag. 'run' (exec a cmd in the sandbox) and 'doctor'-style preconditions are worth adopting; we can skip upload/download/pause/resume/snapshot entirely.
- Snapshot/restore is their headline feature and their biggest complexity sink (stateful disk sizing, IP-reuse collisions, daemon-tracked snapshot IDs). Our threat model (accident not adversary, blast-radius containment) does NOT need it — explicitly out of scope. Backtracking, if ever wanted, is the worktree/git's job, not the sandbox's.
- Their lifecycle is daemon-owned and persistent-by-default, requiring stateful allocators (ip/cid/port/tap) and a state_dir — exactly the concurrency-safe allocation pain our GRADUATION SIGNAL names. We dodge all of it by being ephemeral-by-default (--rm) + label-driven reap + podman labels as truth. Their 20s graceful-then-kill reaper validates our trap+reap backstop design.
- Networking is the clearest contrast: Arrakis defaults to a shared bridge + NAT + outbound internet for guests. We default to --network none and publish only to localhost. Their auto port-forwarding with descriptive labels ('gui','code') is a nice idea, but we keep it orthogonal — emit parseable localhost port info and let ssh/Caddy/Tailscale compose around us; NO --host flag, no built-in forwarding daemon.
- They bake SSH creds into the image (password elara0000). We instead forward the SSH agent so keys never enter the guest (the smolvm idea) — strictly better for our 'secrets stay on host' stance.
- Memory-as-percent-of-host (guest_mem_percentage) is a neat conservative default to borrow conceptually, but we should expose absolute conservative caps via krun.cpus/krun.ram_mib annotations rather than a host-introspecting heuristic — simpler and auditable in one script.
- Their restserver clobbers global host state (deletes all tap*, br0, flushes iptables) on startup — a cautionary tale: our tool must touch ONLY resources it labels as its own and never perform host-global mutations.
- Identity is a freeform VM 'name' tracked in a daemon map + state_dir. We replace this with mandatory podman labels (sandbox.managed-by/.created/.id/.worktree) so the container engine is the registry — no state file, no daemon, queryable via `podman ps --filter label=`.
- License is AGPLv3 + CLA: do not copy code; treat purely as design prior art.

## gh-runner-krunvm (efrecon/gh-runner-krunvm)

**Tier:** inspiration  |  **Language:** POSIX sh (/bin/sh, not bash). ~2557 lines across orchestrator.sh, runner.sh, lib/common.sh, lib/microvm.sh, runner/entrypoint.sh + token.sh.

**Purpose:** Runs self-hosted, ephemeral GitHub Actions runners inside libkrun microVMs on your own host. Each "runner loop" forever creates a pristine microVM, registers it as an ephemeral runner, runs one job, then tears down and recreates. Pure POSIX-shell orchestration over either `podman run --runtime krun` or `krunvm`+`buildah`.


### Verb surface

- `orchestrator.sh` `[-s sleep] [-I] [-l logfd] [-n N] [-p prefix] [-R runtime] [-v] [-h] -- <runner.sh opts>` — TOP-LEVEL ENTRYPOINT. Spawns N background runner loops, each in its own isolation env dir. NOT a multi-verb CLI: it is a single program controlled entirely by short flags. `--` separates orchestrator flags from flags forwarded verbatim to each runner.sh. Installs an EXIT trap that kills all loops and reaps every microVM whose name matches the prefix.
- `runner.sh` `[-c cpus] [-d dns] [-D devdir] [-g github] [-G group] [-i image] [-l logfd] [-L labels] [-m mem] [-M host:vm mount] [-p principal] [-k terminate-secs] [-r repeat] [-R runtime] [-s scope] [-S secret] [-T pat] [-u user] [-U] [-v] [-h] -- <loop-id>` — THE LOOP. `while true`: pull image, mint RUNNER_ID=loop-randomstring, build the microVM, run it (blocks until job done), delete it, repeat. No subcommands; behaviour (ephemeral, repeat count, terminate timer) is all flags/env.
- `runner/entrypoint.sh` `[-e ephemeral] [-E envfile] [-g github] [-G group] [-i id] [-k tokenfile] [-L labels] [-n name] [-p principal] [-s scope] [-t token] [-T pat] [-u user] [-U] [-v] -- <cmd>` — IN-GUEST entrypoint (PID1 via dumb-init). Reads+deletes the .env file, acquires a runner token, configures and registers the .NET runner, optionally starts podman-as-docker daemon, runs the runner as the `runner` user, traps signals to unregister.
- `runner/token.sh` `-g github -l log -p principal -s scope -T pat` — Exchanges a GitHub PAT for a short-lived runner registration token via the GitHub API (curl+jq). Isolated as a subprocess so the PAT never reaches the workflow.
- `microvm_* (lib/microvm.sh internal API)` `microvm_runtime / _list / _run / _wait / _stop / _delete / _pull / _cleanup` — INTERNAL runtime-abstraction layer (sourced, not a CLI). Dispatches each verb to either `podman ...` or `buildah unshare krunvm ...` based on auto-detected runtime. This is the real 'verb vocabulary' worth stealing: list/run/wait/stop/delete/pull/cleanup.

### Lifecycle model
Aggressively ephemeral. Default RUNNER_EPHEMERAL=1 with NO CLI flag to disable it ("the much preferred behaviour is to run ephemeral runners"). Under podman the VM is created with `--rm` so a clean exit self-cleans. Each loop iteration is: pull -> create -> run-to-completion -> delete -> recreate, forever (RUNNER_REPEAT=-1) unless a repeat count or a `.brk` break-file is set. Three teardown layers worth noting: (1) `--rm` clean exit; (2) per-process EXIT/INT/TERM traps in BOTH orchestrator.sh and runner.sh that terminate+delete the VM they own; (3) a PREFIX-driven reaper in the orchestrator's cleanup() that lists ALL microVMs matching `^${PREFIX}-` and deletes each one as a backstop. Graceful in-guest shutdown is signalled out-of-band via sentinel files on a shared mount: `.tkn` (token ready), `.trm` (terminate request, matched against a random RUNNER_SECRET), `.brk` (break the outer loop). Optional `-k SECS` self-terminate timer (`sleep N && cleanup &`).

### Isolation & flags
- Runtime auto-detect: prefers `krun` (=> podman+krun) over `krunvm` (=> buildah). Forced to podman+krun if anything prefixed 'podman' is passed via -R.
- podman path builds: `podman run --runtime krun --rm --tty --name NAME --cpus N --memory Mm --dns IP --entrypoint PATH IMAGE ...`. Uses podman-NATIVE --cpus/--memory, NOT krun annotations (no krun.cpus / krun.ram_mib anywhere in the repo).
- Resource defaults: 2 vCPUs, 1024 MB RAM, DNS 1.1.1.1. NOTES.md flags that >=1024MB is required for dnf to work in-guest.
- Volume mounts: `-v host:vm` lines; under podman, if the mount has no 3rd field it auto-appends `:Z,rw` (SELinux relabel + read-write). So default mount posture is READ-WRITE, NOT read-only.
- NETWORK: no host-side network isolation at all. In-guest, containers are forced to `--network host` (microVM limitation) via a transparent docker->docker.orig CLI shim (base/docker.sh) that injects `--network host` into `run`/`container run`.
- containers.conf in base image sets netns="host" and documents runtime_supports_kvm=["kata","krun"].
- Secret isolation is the headline isolation feature: all RUNNER_* config (incl. PAT) is written to a per-run `.env` file on a shared mount, which entrypoint.sh sources THEN immediately `rm`s, so secrets never hit the process table or persist. `-I` turns this off (documented as a security risk).
- No `/dev/kvm` / KVM-access precondition check exists anywhere. check_command only verifies binaries (krun/podman/buildah) are on PATH.

### Standout features
- Runtime-abstraction layer (lib/microvm.sh): one set of verbs (list/run/wait/stop/delete/pull/cleanup) dispatched to podman OR krunvm. Clean seam for 'engine is fixed but swappable'.
- Self-documenting help: usage() greps the script's own `letter) # comment` lines and reformats them into a help listing, and dumps current PREFIX_ env state (filtering TOKEN). Zero duplication between arg-parser and help text.
- Env-var + flag duality with flags taking precedence; every var is `${VAR:-default}`. Whole tool is scriptable headlessly via env.
- Poor-Man's-Logging (PML): leveled trace/debug/verbose/info/warn/error to a configurable fd, with `[bin] [LVL] [timestamp]` prefix; `sublog()` tails a guest log file and re-stamps each line through the same formatter for unified host-side logs.
- Sentinel-file IPC over a shared mount (.tkn/.trm/.brk) to coordinate host loop <-> in-guest process without a daemon or socket.
- Pure-POSIX portability helpers reimplemented to dodge GNU/macOS differences: abspath (readlink -f), pgrep_P, ps_tree, kill_tree, waitpid, tac via awk.
- Two-layer image build: base OS image (fedora/ubuntu) separate from runner install, so the OS is swappable.

### Gotchas
- set -eu only — NO pipefail (it's /bin/sh, where pipefail isn't POSIX). Pipelines that grep/cut can mask upstream failures. Our bash tool should use the stronger `set -euo pipefail`.
- The PREFIX-based reaper is a substring/regex match (`grep -E ^PREFIX-`). A sloppy prefix could match unrelated containers. Argues for our label-driven reap (podman --filter label=...) being far safer than name matching.
- Traps disarm themselves first (`trap '' EXIT` / `trap - INT TERM EXIT`) to avoid re-entrancy/double-cleanup — important pattern, easy to forget.
- Default volume posture is :Z,rw (read-WRITE + SELinux relabel). The :Z relabel can mutate host-dir SELinux labels — surprising side effect on shared dirs. Our 'read-only by default' stance is the opposite and safer.
- krunvm path needs `buildah unshare` wrapping for EVERYTHING and maintains its own VM->PID file store under a mktemp dir because krunvm has no real `ps`/`wait`; podman path is much simpler. Confirms podman+krun is the cleaner engine.
- No KVM/permission preflight — failures surface late and cryptically. Reinforces our 'doctor is verb zero' decision.
- RUNNER_SECRET defaults to a random string per process; if orchestrator and guest don't share it (isolation off), .trm/.brk termination silently won't match.
- Concurrency: start-up is serialized with a 30s sleep (or token-file wait) between loops — a crude mutex, exactly the 'concurrency-safe allocation' pain our graduation signal warns about.

### Fedora fit
Medium-high as design inspiration, low as packaging. It explicitly documents Fedora as the easiest host install and ships a Dockerfile.base.fedora, and its preferred runtime is precisely our locked stack (podman + krun). But it is a third-party GitHub project (not a Fedora-packaged tool), is written in portable POSIX sh to also support macOS/krunvm/buildah, and predates/ignores the krun annotation interface (krun.cpus/krun.ram_mib) — it drives resources through podman-native flags. So: same engine, same OS bias, but not Fedora-canon and broader/more portable than our single-user single-host scope.

### Lessons for our tool
- Adopt the microvm.sh verb vocabulary for our own function/subcommand naming inspiration: list / run / wait / stop / delete / pull / cleanup are clear, scriptable verbs — but back them with LABEL filters, not name-prefix greps.
- Centralized creation is validated: they funnel every VM through microvm_run() which stamps name/cpus/mem/dns/--rm. Mirror this with our single born-here function that stamps the mandatory label set + safe-by-default flags.
- Use podman-native --cpus/--memory like they do, but ALSO add the krun.cpus/krun.ram_mib annotations our spec calls for; note this repo proves the bare --runtime krun + --cpus/--memory path works without annotations.
- Their three-layer teardown maps almost exactly onto our locked plan (--rm + traps + reaper). Steal the detail of DISARMING the trap first (`trap '' EXIT`) to prevent double-cleanup, and reaping as an idempotent backstop.
- ADD the KVM/doctor preflight they lack: check /dev/kvm access, `crun --version | grep +LIBKRUN`, and crun-krun presence BEFORE any run. Their late cryptic failures are the cautionary tale.
- Replace name-prefix matching with podman `--filter label=sandbox.managed-by=...`; their `grep ^PREFIX-` reaper is the unsafe version of our label-driven reap.
- Steal the self-documenting usage() (grep the script's own option-comments) and the leveled-logging-to-fd pattern for a clean, auditable single-file script.
- Their secret-isolation-via-ephemeral-.env-file (source then rm) generalizes our SSH-agent-forwarding goal: keep credentials out of the guest's process table and filesystem; never pass secrets on the command line.
- Use `set -euo pipefail` (we're bash, not their constrained /bin/sh) — they couldn't, we can and should.
- Their 30s-sleep startup mutex is exactly the 'concurrency-safe allocation' graduation pain; if we ever need parallel sandboxes with port allocation, that's the signal to leave bash.
- For machine-readable output, note they only emit human logs; our --json/stable-columns mode is a real differentiator — derive it straight from `podman ps --filter label --format`.
- The docker.sh CLI-shim trick (transparently inject a flag into a wrapped binary) is a clean pattern if we ever need to enforce flags on a tool the workload calls.

## textcortex/claude-code-sandbox

**Tier:** inspiration  |  **Language:** TypeScript (Node.js CLI, Commander.js, dockerode)

**Purpose:** A CLI that runs the Claude Code AGENT itself inside a Docker (or Podman) container with git integration, so the agent can run with --dangerously-skip-permissions safely. It sandboxes the harness, not the workload — a different layer than our tool, but a rich source of config/lifecycle/verb-surface UX prior art.


### Verb surface

- `(default / no subcommand)` `[--shell claude|bash]` — Bare `claude-sandbox` starts a new sandbox container from cwd and launches a web UI. Equivalent to `start` but with includeUntracked forced false. Lesson: a bare invocation = create+run is convenient but conflates 'create' with the noun; we prefer explicit verbs.
- `start` `-c/--config <path>, -n/--name <prefix>, --no-push, --no-create-pr, --include-untracked, -b/--branch <branch>, --remote-branch <branch>, --pr <number>, --shell claude|bash` — Explicitly create+start a new container. Generates a branch (claude/<date>-<epoch>) unless one is given; can check out a PR or remote branch INSIDE the container. This is the primary 'birth' verb.
- `attach` `[container-id]` — Attach to an existing container; if no id given, shows an interactive inquirer picker over containers whose name contains 'claude-code-sandbox'. Always opens the web UI rather than a raw TTY.
- `list (alias: ls)` `-a/--all (include stopped)` — List managed containers with short-id, name, colored state, status. Filters by NAME SUBSTRING 'claude-code-sandbox' (no label-based selection — a weakness).
- `stop` `[container-id], -a/--all` — Stop one container (interactive pick if id omitted, running-only) or all running managed containers. Does NOT remove.
- `logs` `[container-id], -f/--follow, -n/--tail <lines> (default 50)` — Stream container logs; demuxes Docker's multiplexed stdout/stderr. Interactive pick if id omitted.
- `clean` `-f/--force` — Remove STOPPED managed containers; with --force also stops+removes running ones. The gentle reaper.
- `purge` `-y/--yes` — Stop (5s timeout) AND remove ALL managed containers; prompts for confirmation unless -y. The nuclear reaper. Also exposed as npm script `purge-containers` that nukes containers + image by ancestor filter.
- `config` `-p/--path <path> (default ./claude-sandbox.config.json)` — Load and pretty-print the effective merged JSON config. A read-only introspection verb — good prior art for our --json/doctor surface.

### Lifecycle model
PERSISTENT-by-default, manually reaped. Containers are created with HostConfig.AutoRemove:false and a name `${prefix}-${Date.now()}`; nothing auto-removes on exit. The CLI process stays alive (`await new Promise(()=>{})`) to keep a session/web-UI open; on the in-process commit-review 'exit' action it calls cleanup() which stop()+remove()s containers it tracked in an in-memory Map. There is NO label/state file and NO background reaper — selection is purely by container-NAME substring 'claude-code-sandbox'. Cleanup is entirely operator-driven via clean/stop/purge verbs (a graduated severity ladder: stop < clean < purge). In-memory tracking is lost if the CLI dies, so orphaned containers are found only by name-scan. This is the OPPOSITE of our ephemeral --rm-by-default + trap + label-reap model, and shows the failure mode we are avoiding.

### Isolation & flags
- NetworkMode: 'bridge' (full outbound network ON by default — the agent needs to reach Anthropic/GitHub APIs). No network-none option. We invert this: --network none default.
- Workspace is NOT bind-mounted: files are copied IN via `git archive HEAD` + .git tarball through container.putArchive. Deliberate isolation choice so the agent's changes never touch the host tree directly — instead a 'shadow repository' is rsync'd back. Contrast with our worktree-per-sandbox bind approach.
- Credentials mounted/injected read-only-ish: ANTHROPIC_API_KEY / Bedrock / Vertex / OAuth token discovered on host and passed as ENV; GitHub token discovered (env, GH_TOKEN, or `gh auth token`) and injected as GITHUB_TOKEN + wired into git via url.insteadOf rewrite. Comment in code: 'NO SSH mounting - we use GitHub tokens instead' — they explicitly avoid mounting ~/.ssh (we instead forward ssh-agent so keys never enter guest).
- Custom `mounts[]` schema: {source,target,readonly} with $HOME/$VAR expansion; readonly appends ':ro'. Legacy `volumes[]` raw strings also supported.
- NO resource caps at all — no --memory/--cpus/pids equivalent anywhere. A gap vs our krun.cpus/krun.ram_mib annotation defaults.
- Container runs as non-root user 'claude' but with NOPASSWD sudo ALL (so isolation relies entirely on the container boundary, not in-guest privilege).
- env CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1 always set; optional MAX_THINKING_TOKENS, BASH_MAX_TIMEOUT_MS from config.
- Container is created with Tty+OpenStdin+AttachStdin for interactive attach; WorkingDir /workspace; Cmd ['/bin/bash','-l'].

### Standout features
- Podman auto-detection without any --runtime flag: docker-config.ts probes a fixed list of socket paths in order — /var/run/docker.sock, $XDG_RUNTIME_DIR/podman/podman.sock, /run/user/<uid>/podman/podman.sock, /run/podman/podman.sock — honors DOCKER_HOST first, then config dockerSocketPath; isPodman() = socket path contains 'podman'. Zero-config engine selection.
- Credential DISCOVERY ladder (credentials.ts): tries env, ~/.claude.json, Bedrock/Vertex env flags, OAuth token files in 3 locations, macOS Keychain. Treats Claude creds as OPTIONAL (warns, lets you log in inside container) but is strict-ish on surfacing GitHub creds.
- Files copied via `git archive` so only tracked (+ optionally --include-untracked, gitignore-respecting) files enter the sandbox — clean, reproducible, no host coupling.
- 'Shadow repository' rsync-back with documented rule precedence (git-tracked beats .gitignore; built-in excludes .git/node_modules/__pycache__) so committed data files are never lost on sync.
- Real-time git-monitor that detects the agent's commits and pops an interactive review (do nothing / push / push+PR / exit) — autoPush/autoCreatePR config gates this.
- In-container git wrapper concept (commented-out) that blocks `git checkout`/`switch` after the initial branch to protect main — directly relevant to our 'guard destructive git inside' requirement.
- Graduated cleanup verbs (stop/clean/purge) with confirmation + -y bypass; purge prints exactly what it will remove before acting.
- config example file is small and flat — good readable schema for documenting defaults.

### Gotchas
- Selection by NAME SUBSTRING, not labels: any container whose name happens to contain 'claude-code-sandbox' is swept by list/stop/clean/purge. Fragile and collision-prone. We MUST select by our managed-by label instead.
- No state persistence + in-memory container Map: if the long-lived CLI process crashes, tracked containers are orphaned and only re-findable by name scan; no reaper backstop. This is precisely the gap our trap+label-reap design closes.
- Persistent-by-default (AutoRemove:false) means leaks accumulate silently until the user runs clean/purge.
- Full bridge network by default — appropriate for an agent that must hit APIs, but a footgun for a blast-radius tool; not safe-by-default for workload isolation.
- NOPASSWD sudo ALL inside the guest means in-container compromise = full container root; fine only because the container is the trust boundary.
- setupCommands 'set -e' handling is a hack (it string-matches whether 'set -e' is an element of the array to decide whether to fail) — brittle config semantics.
- Web-UI / socket.io / express stack is heavy for what is essentially terminal attach; lots of deps for a sandbox tool.
- macOS-specific branches everywhere (Keychain, --no-xattrs/--no-fflags tar) — portability tax baked into the copy path.

### Fedora fit
Not Fedora-canon at all. It's an npm/TypeScript tool that builds its own Ubuntu 22.04 image (apt, NodeSource, gh keyring) and talks to the Docker/Podman SOCKET via dockerode — i.e. it requires the Podman API socket service running (rootless or system), not plain `podman run`. It DOES detect rootless Podman sockets and even prints 'Detected Podman socket', so it interoperates with Fedora's Podman, but it assumes the daemon/socket model we are deliberately rejecting (we use daemonless `podman run --runtime=krun` directly, no socket).

### Lessons for our tool
- Verb-naming prior art: their surface is start / attach / list(ls) / stop / logs / clean / purge / config. The stop<clean<purge severity ladder is good UX — consider mirroring (e.g. our `reap` as the nuclear backstop). 'config' as a read-only introspection verb pairs well with our planned 'doctor' (verb zero) and --json output.
- Selection-by-name is their core weakness — it validates our LOCKED decision to select exclusively by a mandatory managed-by label set. Never filter sandboxes by name substring.
- Their persistent-by-default + in-memory tracking + no reaper is the exact failure mode our (--rm default) + (trap INT/TERM/EXIT) + (label-driven reap backstop) three-layer design exists to prevent. Strong confirmation of our locked cleanup model.
- SSH: they explicitly chose 'no SSH mounting, use GitHub tokens' and rewrite git remotes via url.insteadOf. We instead forward ssh-agent (smolvm idea) so keys never enter the guest — keep that, and steal their `git config url.<token>.insteadOf` trick only if token-based auth is ever needed, but prefer agent forwarding.
- Git safety: their commented-out in-container git wrapper that blocks checkout/switch after the first branch is a concrete pattern for our 'guard destructive git inside the worktree' requirement.
- They COPY a `git archive HEAD` snapshot in rather than bind-mounting the repo, decoupling the agent from the host .git, then rsync a shadow repo back. We instead CREATE a worktree and bind it — but note their motivation (don't let clumsy agent code corrupt the real tree) matches our threat model; our worktree-per-sandbox + destructive-git guard achieves the same containment with less machinery.
- Engine detection: their ordered socket-probe (DOCKER_HOST > config path > known socket paths > isPodman by substring) is elegant, but it's socket-based. Our doctor should instead verify the daemonless path: KVM access, `crun --version | grep LIBKRUN`, crun-krun present, libkrun/libkrunfw — a precondition check, not a socket probe.
- Add the resource caps they lack: they set NO --memory/--cpus/pids limits. Our centralized birth function should always stamp conservative krun.cpus / krun.ram_mib annotations.
- Network default: they need bridge because they sandbox the agent (must reach APIs). We sandbox the WORKLOAD, so --network none default is correct; only open ports via explicit publish-to-localhost. Their choice reinforces that network policy follows what-you-sandbox.
- Config-file vs flags: they use a JSON config file (claude-sandbox.config.json) merged with flags. For our single-user flag-preceded bash script, prefer flags + podman labels as the only source of truth; avoid a state/config file, which they also (mostly) avoid for runtime state.
- Their `config` verb just pretty-prints merged config — cheap, high-value introspection. Implement an equivalent that emits stable columns / --json so ssh/Caddy can compose around our published localhost ports.

## sbx-releases (Docker Sandboxes / `sbx`)

**Tier:** inspiration  |  **Language:** Go (closed-source binary; this repo is docs + release artifacts only)

**Purpose:** Docker's proprietary "safe environment for agents" tool. `sbx` spins up per-workspace microVMs (each with its OWN Docker daemon, filesystem, network) so coding agents (Claude Code, Codex, Gemini CLI, OpenCode) can run in "YOLO mode" without permission prompts and without risking the host. This repo itself is a thin release/distribution + docs shell; the real CLI and policy engine are external.


### Verb surface

- `sbx login` `[--username U] [--password-stdin]` — Browser-based Docker OAuth; MANDATORY before any sandbox runs. On first login it interactively prompts for a default network policy (Open / Balanced / Locked Down). Non-interactive variant via --username/--password-stdin.
- `sbx run` `<agent> [workspace...] [--clone <git-url>] [--name <name>]` — Create a sandbox AND attach to the agent in one step (e.g. `sbx run claude`, `sbx run claude ~/project`). First/primary positional is the agent (claude, codex, gemini, opencode). Extra path args are additional mounts. Re-running the same workspace path RECONNECTS to the existing sandbox instead of making a new one.
- `sbx create` `<agent> <workspace> [--clone] [--name]` — Like run but background/detached — creates the sandbox without attaching. Requires an EXPLICIT workspace path (e.g. `sbx create claude .`).
- `sbx ls` `` — List sandboxes with status, CPU, memory, and port mappings.
- `sbx stop` `<name>` — Pause/temporarily stop a sandbox (state kept; not deleted).
- `sbx rm` `<name>` — Delete a sandbox entirely. WARNING: removing a clone-mode sandbox drops its in-sandbox git clone; unfetched commits are lost.
- `sbx exec` `[-it] <name> <cmd> (e.g. bash)` — Run a command / get a shell inside a running sandbox.
- `sbx cp` `<src> <name>:<dest>  (or reverse)` — Copy files between host and sandbox.
- `sbx ports` `<name> --publish HOST:GUEST | --unpublish ...` — Forward (publish) or unpublish ports from host to sandbox services after the fact.
- `sbx policy allow` `network [-g] <domain>` — Add an allow rule for network egress to a domain. `-g` makes it global (all sandboxes); without -g it scopes to one sandbox. e.g. `sbx policy allow network -g api.anthropic.com`.
- `sbx policy deny` `network [-g] <domain>` — Add a deny rule for a domain. e.g. `sbx policy deny network -g ads.example.com`. Deny ALWAYS beats allow regardless of specificity or scope.
- `sbx policy reset` `` — Reset the local policy store back to defaults. (Org-managed policies override local ones and cannot be reset away.)

### Lifecycle model
PERSISTENT by default — the opposite of our chosen model. Sandboxes survive agent exit; `sbx run <same-workspace>` reconnects rather than recreating. Lifecycle verbs are explicit and manual: stop (pause), rm (destroy). There is NO automatic reaper / TTL / ephemeral --rm behavior documented — the human is expected to `sbx rm` to reclaim disk (and disk grows because each VM keeps its own Docker daemon state, image cache, and built layers). Identity is keyed on workspace path (+ optional --name), which is how reconnect works.

### Isolation & flags
- Engine: microVM per sandbox (full hypervisor isolation), each with its OWN Docker daemon, image cache, and package installs — fully isolated from other sandboxes (no shared layers).
- Linux requires KVM: install docs run `sudo usermod -aG kvm $USER` + `newgrp kvm` — a direct precedent for our 'doctor' KVM-access precondition check.
- Filesystem: workspaces are mounted via direct passthrough at their ABSOLUTE host paths; changes propagate instantly (no sync). First path = primary workspace.
- Mount mode: multiple workspaces allowed; `:ro` suffix marks a mount read-only, e.g. `sbx run claude ~/project-a ~/shared-libs:ro ~/docs:ro`. Extra workspaces are mounted directly (never cloned).
- Git: --clone mode does an in-sandbox git clone (isolating the agent from the host repo) instead of a passthrough mount.
- Network egress: ALL outbound HTTP/HTTPS routes through an HTTP/HTTPS proxy ON THE HOST that enforces policy and injects credentials. Default = blocked unless an explicit allow rule matches. UDP and ICMP are PERMANENTLY blocked; TCP can be unblocked via explicit rules only.
- Three network policy presets chosen at first login: Open (all traffic allowed), Balanced (default-deny + baseline allowlist of AI provider APIs, package managers, code hosts, container registries, common cloud services), Locked Down (everything blocked incl. api.anthropic.com).
- Policy precedence: org/admin-managed policies > local; deny > allow.
- Resource caps (CPU/memory) exist and are surfaced in `sbx ls`, but specific default limits/flags are not documented publicly.
- Ports: nothing reachable from host until explicitly published via `sbx ports --publish HOST:GUEST` (default-closed network surface).

### Standout features
- The egress-policy verb model is the headline takeaway: `sbx policy allow|deny network [-g] <domain>` with three named presets (Open / Balanced / Locked Down) and a clear 'deny beats allow' rule.
- Per-workspace-path identity enabling automatic reconnect (no manual ID juggling).
- `:ro` mount suffix for read-only extra workspaces — a clean, learnable syntax.
- Clone-mode (--clone) explicitly isolates the agent's git from the host repo.
- Host-side egress proxy that both enforces policy AND injects credentials (so secrets aren't baked into the guest).
- run = create+attach vs create = detached background — a clean attached/detached split.
- Centralized org admin console for fleet-wide network + filesystem policy.

### Gotchas
- PROPRIETARY (Docker Inc., closed source) and requires MANDATORY browser OAuth login (`sbx login`) — disqualifying for a single-user, no-login, Fedora-canon tool. This is the strongest 'do NOT emulate' signal.
- Persistent-by-default + no reaper means disk silently grows (each VM hoards its own Docker daemon state + image cache); cleanup is entirely manual `sbx rm`.
- `sbx rm` on a clone-mode sandbox SILENTLY destroys unfetched commits — a data-loss footgun our git guard must avoid.
- Heavy: a full VM PLUS a nested Docker daemon per sandbox = high overhead vs our single podman/krun microVM.
- Distribution is non-Fedora-first: Homebrew/WinGet/APT(Ubuntu)/Rocky8 RPM/Nightly — no native Fedora package.
- Linux egress only proxies HTTP/HTTPS cleanly; UDP/ICMP are hard-blocked, which can surprise workloads (DNS-over-UDP, ping).

### Fedora fit
LOW / anti-pattern for canon. Closed-source Docker proprietary binary, distributed via Homebrew/WinGet/APT/Rocky-RPM (no Fedora package, not in Fedora repos), runs its own Docker daemon, and requires a mandatory Docker-account OAuth login. The only Fedora-aligned overlap is the KVM-group requirement. Treat purely as design inspiration, not as something to package or depend on.

### Lessons for our tool
- VERB NAMING: their surface maps cleanly onto ours — adopt `ls` (not `list`), `rm` (not `delete`), `exec`, `run`, plus a detached `create`. Their run=create+attach / create=detached split is a clean precedent if we ever want a background mode; for v1 keep `run` (ephemeral) as the core verb.
- EGRESS at the firewalld tier: emulate the THREE NAMED PRESETS (open / balanced / locked-down) and the simple `allow|deny <domain>` grammar with a hard rule 'deny beats allow'. For our accident-not-adversary model, default `--network none` ~= their Locked Down; a 'balanced' preset (allow package mirrors + the agent's model API) is the useful middle and worth pre-baking as a named flag value.
- MOUNTS: steal the `path:ro` suffix syntax for read-only extra mounts — terse, learnable, and matches our 'read-only where possible' default. Mount at absolute host paths so the agent sees familiar paths.
- GIT/WORKTREE: their --clone (in-sandbox clone isolating the agent from host .git) plus their documented 'rm drops unfetched commits' footgun validates BOTH our worktree-per-sandbox decision AND our requirement to guard destructive git — but we should CREATE the worktree on the host (theirs orphans commits inside the VM). Never silently destroy a worktree with unpushed commits on reap.
- LIFECYCLE — DIVERGE DELIBERATELY: they are persistent + manual-only cleanup and it causes silent disk bloat. Our ephemeral-by-default + 3-layer cleanup (--rm, trap, label-reaper) is the better posture; cite them as the cautionary 'no reaper' case. Persistence stays an explicit opt-in verb/flag as decided.
- DOCTOR/PRECONDITION: their `usermod -aG kvm` install step confirms KVM-group membership is a real first-run failure mode — our verb-zero doctor should check exactly this (KVM access) alongside crun+LIBKRUN.
- CREDENTIALS: their host-side proxy injects creds so secrets never live in the guest — same spirit as our SSH-agent forwarding; keep secrets on the host, forward, never copy in.
- PORTS: default-closed network surface with explicit publish (their `sbx ports --publish HOST:GUEST`) matches our 'publish predictably to localhost, emit parseable output' plan; do NOT add a --host flag (stay orthogonal to Tailscale), just publish to localhost.
- OUTPUT: `sbx ls` showing status/CPU/mem/ports is the column set users expect — provide a stable-columns + --json mode covering id, status, ports, worktree.
- DISTRIBUTION CONTRAST: their mandatory login + closed binary + non-Fedora packaging is precisely what we are reacting against — our value prop is single-file auditable bash, no login, no daemon, Fedora packages only.

### Lessons for our tool
- Adopt the field's canonical access-tier vocabulary: Codex CLI's `read-only` / `workspace-write` / `danger-full-access` is the most-cited naming for escalating filesystem/network access — a strong template for our isolation tier flag, far better than inventing new words.
- Our two-mode lifecycle (ephemeral --rm default + explicit persist opt-in) directly mirrors microsandbox's persistent-mode + `msx` ephemeral-mode split — prior art validates the design; consider an `msx`-like short verb for the throwaway path.
- Deny-by-default network is the dominant safe default across the serious tools (Cleanroom, treebeard, Greywall, srt). Our `--network none` default is squarely canonical; treat any network opening as the explicit, labelled exception.
- Worktree-per-sandbox where the TOOL creates the worktree is well-trodden (container-use, Conductor, packnplay, treebeard). treebeard pairs it with CoW + network gating — a good reference design to read next for the create-the-worktree-and-guard-destructive-git behavior.
- Keys-out-of-guest via a proxy/forward is an established hardening move (Cleanroom credential proxy, srt proxy) — reinforces our SSH-agent-forwarding decision; frame it as 'no secrets cross the boundary'.
- We sit precisely in the microVM 'hardware-virtualized, container-fast' lane the doc endorses as the strong-isolation-yet-fast sweet spot — good positioning language for our SKILL.md.
- Read the actual repos for real verb surfaces: this doc gives ZERO commands. The highest-value next reads for verb/lifecycle/flag prior art are microsandbox (msx, persistent mode, MCP), treebeard (worktree + CoW + net gating), Cleanroom (deny-by-default egress, credential proxy), and codex-container-sandbox / packnplay (Podman repo-only bind-mount wrappers, the closest siblings to our design).
- The graduation signal in the doc's framing: when single-file bash isolation guarantees become hard to assert, microsandbox/libkrun-native is the named next step — consistent with our stated graduation plan.
